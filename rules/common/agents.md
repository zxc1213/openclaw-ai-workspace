# Agent 编排规则

## 可用 Agent 角色

| Agent | 用途 | 触发时机 |
|-------|------|----------|
| coder | 主力编码 | 一般开发任务 |
| reviewer | 代码审查 | 写完代码后 |
| researcher | 调研分析 | 技术调研、方案对比 |
| writer | 文档撰写 | 技术文档、方案 |
| analyst | 数据分析 | 数据处理 |
| designer | 设计/原型 | UI/UX 设计 |
| cron-worker | 定时任务 | Heartbeat/Cron 执行 |

## 编排策略

### 并行
```
独立任务 → 同时 spawn 多个 subagent
例: 安全审查 + 性能审查 + 测试覆盖检查
```

### 串行
```
依赖任务 → 上游完成再启动下游
例: 调研 → 方案设计 → 实现 → 审查
```

### 多角色审查（复杂问题）
- 事实审查者：验证逻辑正确性
- 资深工程师：评估架构合理性
- 安全专家：检查安全隐患

## 任务派发规范

所有 subagent 任务必须包含四要素：
1. **Goal** — 一句话说清楚做什么、产出什么
2. **Context** — 关键背景（<5 行）
3. **Constraints** — 硬性约束（≤5 条）
4. **Done 标准** — 可检查的完成条件
