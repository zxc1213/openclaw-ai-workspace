# PR Review Checklist

## 审查流程

### Step 1: 获取变更范围

```bash
# 获取 diff（审查 PR 时）
git diff origin/main...HEAD --stat

# 或审查指定 PR
gh pr diff <PR#>

# 或审查指定 commit
git show <commit-hash> --stat
```

快速扫描变更文件列表，形成整体印象。

### Step 2: 逐项审查（按顺序）

按以下清单逐项检查，每项标注 ✅/⚠️/❌ 和简短说明。

---

## 审查清单

### 🔴 必须项（任一 ❌ 则建议拒绝）

| # | 检查项 | 说明 |
|---|--------|------|
| 1 | **功能正确性** | 代码是否实现了 PR 描述的功能？逻辑是否正确？ |
| 2 | **无引入新 bug** | 边界条件是否处理？空值/异常是否考虑？ |
| 3 | **无安全漏洞** | SQL 注入、XSS、硬编码密钥、不安全的反序列化、权限绕过 |
| 4 | **无破坏性变更** | 是否有 API 不兼容改动？数据库 schema 变更是否可回滚？ |

### 🟡 重要项（影响代码质量）

| # | 检查项 | 说明 |
|---|--------|------|
| 5 | **代码可读性** | 变量命名清晰？复杂逻辑有注释？函数职责单一？ |
| 6 | **错误处理** | 异常是否被正确捕获和处理？错误信息是否有意义？ |
| 7 | **性能影响** | 是否引入 N+1 查询？大循环？不必要的内存分配？ |
| 8 | **测试覆盖** | 新代码有对应测试？修改的代码现有测试是否通过？ |
| 9 | **日志与监控** | 关键操作有日志？异常路径有日志？日志级别恰当？ |

### 🟢 建议项（锦上添花）

| # | 检查项 | 说明 |
|---|--------|------|
| 10 | **代码风格一致** | 与项目现有风格一致？格式化是否统一？ |
| 11 | **DRY 原则** | 是否有可抽取的公共逻辑？ |
| 12 | **文档更新** | API 文档、README、注释是否需要同步更新？ |
| 13 | **提交信息规范** | commit message 清晰？原子提交？ |

## 审查输出模板

```markdown
## PR Review: [PR标题/编号]

### 概览
- 变更文件: X 个
- 新增行: +XX / 删除行: -XX
- 整体评估: ✅ Approve / ⚠️ Request Changes / ❌ Reject

### 审查结果

| # | 检查项 | 结果 | 说明 |
|---|--------|------|------|
| 1 | 功能正确性 | ✅ | ... |
| 2 | 无新 bug | ⚠️ | ... |
| 3 | 安全漏洞 | ✅ | ... |
| ... | ... | ... | ... |

### 关键问题（必须修复）
1. [文件:行号] 问题描述 → 建议修复方式

### 建议改进（可选）
1. [文件:行号] 优化建议

### 总结
[一段话总结审查结论]
```

## 常见问题速查

### SQL 注入
```bash
# 搜索 MyBatis ${} 使用（Java 项目）
grep -rn '\${' --include="*.xml" mapper/
```

### 硬编码密钥
```bash
grep -rn 'password\|secret\|api_key\|token' --include="*.java\|*.yml\|*.properties" src/ | grep -v 'placeholder\|example\|TODO'
```

### 未处理的异常
```bash
grep -rn 'catch.*Exception.*{' --include="*.java" src/ | grep -v 'log\|logger\|throw'
```

## 使用方式

### 作为 subagent 审查

```markdown
## 任务：审查 PR #XX

### Goal
审查 PR #XX 的代码变更，按审查清单逐项检查，输出结构化审查报告。

### Context
- 仓库: [repo-url]
- PR: [pr-url]
- 变更范围: [简述]

### Constraints
1. 只审查，不修改代码
2. 使用审查输出模板
3. 上下文预算: 500-800 token

### Done 标准
1. 输出完整审查报告（含概览 + 逐项结果 + 关键问题 + 总结）
2. 明确给出 Approve / Request Changes / Reject 结论
```

### 主 agent 快速审查

当用户发来代码片段或 diff 要求审查时，直接按清单检查并输出结论。

## 反模式

❌ 只看代码不看测试（测试和实现同等重要）
❌ 审查时顺手改代码（审查 ≠ 编码，发现问题提建议即可）
❌ 只关注风格忽略逻辑（正确性 > 可读性 > 风格）
❌ 审查报告太长或太短（有问题的说清楚，没问题的简略带过）
