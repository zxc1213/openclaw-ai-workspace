# Web Access Skill（OpenClaw 版）

## 核心理念

**像人一样思考，目标驱动而非步骤驱动。**

拿到任务先明确「什么算完成」，然后带着目标进入，边看边判断。遇到阻碍就解决，发现方向错了立即调整。不提前规划所有步骤，不过度依赖固有印象。

---

## 三层通道调度

根据任务性质选择最合适的工具层。从轻到重，逐步升级。

### 第一层：web_search（信息发现）

**适用场景：**
- 搜索摘要、关键词结果
- 发现信息来源和 URL
- 概况性了解一个主题

**用法：**
- 直接使用 `web_search` 工具
- 搜索结果中筛选可信来源（官方、权威媒体）
- 拿到 URL 后评估是否需要进入下一层

### 第二层：web_fetch（定向提取）

**适用场景：**
- URL 已知，需要从页面提取特定信息
- 文章、文档、博客等以正文为核心的页面
- 不需要登录、不需要 JS 渲染的静态内容

**用法：**
- 直接使用 `web_fetch` 工具
- `extractMode: "markdown"` 适合文章/文档
- `extractMode: "text"` 适合纯文本提取
- 设置 `maxChars` 控制返回量，避免浪费 token

**不适用：**
- 需要 JS 渲染的 SPA 页面
- 需要登录态的内容
- 需要交互操作（点击、翻页、滚动加载）

### 第三层：browser（真实浏览器环境）

**适用场景：**
- 需要 JS 渲染的动态页面（SPA、懒加载）
- 需要登录态的内容
- 需要交互操作（点击、翻页、表单填写）
- 反爬严格的平台（小红书、微信公众号等）
- web_search + web_fetch 多次尝试后仍无法获取的内容

**用法：**
- `browser action=snapshot` — 获取页面结构和可交互元素
- `browser action=navigate` + `url` — 导航到目标页面
- `browser action=act` — 点击、填写、滚动等操作
- `browser action=screenshot` — 截图用于视觉分析
- `browser action=eval` — 执行 JS 提取数据
- `browser action=open` — 打开新 tab

**配置选择：**
- `profile="openclaw"` — 隔离的无状态浏览器（默认）
- `profile="windows"` — 连接用户日常 Chrome（携带登录态）
- 使用用户 Chrome 前确认目标内容是否在隔离浏览器中可获取

### Jina（可选预处理层）

将网页转为 Markdown，大幅节省 token。适合文章/文档，不适合数据面板/商品页。

**用法：** 在 `web_fetch` 的 URL 前加 `https://r.jina.ai/` 前缀（去掉原 http://）
- 例：`https://r.jina.ai/https://example.com/article`
- 限 20 RPM

### 调度决策表

| 场景 | 推荐工具 | 原因 |
|------|----------|------|
| "搜一下 xxx" | `web_search` | 纯搜索 |
| "读一下这个 URL" | `web_fetch` | URL 已知，静态内容 |
| "看看这个页面有什么" | `browser snapshot` | 需要完整页面结构 |
| "登录后查看" | `browser` + 用户 Chrome | 需要登录态 |
| "去小红书搜 xxx" | `browser` + 用户 Chrome | 反爬平台 |
| "对比这 5 个产品" | `web_search` → `web_fetch` + 并行 | 多目标调研 |
| "下载这个页面上的图片" | `browser eval` 提取 URL → `exec` 下载 | 需要动态 DOM |
| "填表提交" | `browser act` | 需要交互 |

## 浏览哲学

**① 明确目标** — 什么算完成了？需要什么信息、执行什么操作、达到什么结果？

**② 选择起点** — 根据任务性质选最可能直达的方式。一次成功最好；不成功在③调整。

**③ 过程校验** — 每步结果都是证据。用结果对照成功标准，更新对目标的判断。方向错了立即调整，不在同一方式上反复重试。

**④ 完成判断** — 确认完成才停止，但不过度操作。

## 技术事实

- 页面中存在大量已加载但未展示的内容（轮播非当前帧、折叠区块、懒加载占位元素），它们存在于 DOM 中但对用户不可见。以数据结构为单位思考可以直接触达。
- `/scroll` 到底部触发懒加载，使未进入视口的图片完成加载。提取图片 URL 前应先滚动。
- 短时间内密集打开大量页面可能触发反爬风控。控制并发，适当延迟。
- 平台返回的「内容不存在」等提示不一定是真的——也可能是 URL 缺参数或触发反爬。
- 站点内交互产生的链接自带完整上下文（token、参数），手动构造 URL 可能缺失隐式必要参数。
- 程序化方式（构造 URL / eval DOM）快但可能触发反爬；GUI 交互（点击/填写/滚动）慢但确定性最高。根据对平台的了解灵活选择。

## 并行调研：子 Agent 分治

任务包含多个**独立**调研目标时，分发子 Agent 并行执行，而非串行处理。

**好处：**
- 速度：多子 Agent 并行，总耗时约等于单个子任务时长
- 上下文保护：抓取内容不进入主 Agent 上下文，节省 token

**分治判断：**

| 适合分治 | 不适合分治 |
|----------|-----------|
| 目标相互独立，结果互不依赖 | 目标有依赖关系 |
| 每个子任务量足够大 | 简单单页查询 |
| 需要 CDP 浏览器或长时间运行 | 几次 web_search 就能完成 |

**子 Agent Prompt 写法：**
- 说清楚**要什么**，不限定**怎么做**
- 用目标导向动词（「获取」「调研」「了解」），避免暗示手段的动词（「搜索」「抓取」「爬取」）
- 每个子 Agent prompt 包含 web-access skill 加载指令

## 信息核实原则

核实目标是**一手来源**，而非更多二手报道。

| 信息类型 | 一手来源 |
|----------|---------|
| 政策/法规 | 发布机构官网 |
| 企业公告 | 公司官方新闻页 |
| 学术声明 | 原始论文/机构官网 |
| 工具能力/用法 | 官方文档、源码 |

搜索引擎是信息发现入口，不是证明工具。找到来源后直接读取原文。

## 站点经验库

按域名存储操作经验（URL 模式、平台特征、已知陷阱），跨 session 复用。

### 读取经验

确定目标网站后，检查 `references/site-patterns/{domain}.md` 是否存在：
```bash
ls skills/web-access/references/site-patterns/
```

有匹配文件则读取，获取先验知识。经验当作「可能有效的提示」而非「保证正确的事实」。按经验操作失败时回退通用模式并更新经验文件。

### 写入经验

CDP 操作成功完成后，若发现了有必要记录的新站点或新模式，主动写入。

**文件格式：**
```markdown
---
domain: example.com
aliases: [示例, Example]
updated: 2026-04-06
---

## 平台特征
架构、反爬行为、登录需求、内容加载方式等事实

## 有效模式
已验证的 URL 模式、操作策略、选择器

## 已知陷阱
什么会失败以及为什么
```
