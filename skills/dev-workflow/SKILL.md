# Dev Workflow — 标准化开发工作流

把一个开发任务拆成可控步骤，每步可验证、可跟踪、可中断恢复。

## 全流程

```
接到开发任务
    ↓
Phase 0: 需求分析 + 任务清单创建（本地 + 飞书）
    ↓
Phase 1: 项目初始化 + Git 初始化
    ↓
Phase 2~N: 逐任务执行（每完成一个 → git commit + 标记完成 + 通知用户）
    ↓
Phase Last: 集成验证 + 最终总结
```

---

## Phase 0: 需求分析 + 任务清单

**目标**: 理解需求、拆分任务、建立跟踪。

### 0.1 需求澄清

如果需求不清晰，先问用户：
- 核心功能是什么？MVP 包含哪些？
- 技术栈偏好？参考项目？
- 有没有特殊的约束（平台、性能、安全）？

### 0.2 创建任务清单（双通道）

**本地文件** `research/{project-name}-tasks.md`：
```markdown
# {Project Name} 任务清单

> 创建时间: {date}
> 项目路径: {path}
> 参考资源: {url/repo}

## 任务列表

| # | 任务 | 状态 | 优先级 | 预估 | Git |
|---|------|------|--------|------|-----|
| 1 | 项目初始化 + 脚手架 | ⬜ | P0 | 5min | |
| 2 | 核心数据层（类型 + API） | ⬜ | P0 | 15min | |
| 3 | UI 主框架 + 布局 | ⬜ | P0 | 20min | |
| ... | ... | ... | ... | ... | ... |

## 状态说明
- ⬜ 待开始  🔄 进行中  ✅ 已完成  ⏭️ 跳过  ❌ 失败

## 备注
- 每完成一个任务 → git commit → 更新状态
- coding agent 完成后 → 验证 → 通知用户
```

**飞书任务** — 创建总任务：
- 标题: `🔧 开发: {Project Name}`
- 描述: 项目路径 + 一句话目标
- 子任务: 从任务清单中创建（每条子任务对应一个任务项）

### 0.3 任务拆分原则

- 每个任务**独立可验证**（做完能跑/能看/能测）
- **最小可用优先**：先搭骨架，再填血肉
- 依赖关系明确的按顺序，无依赖的可并行
- 每个任务控制在 coding agent 一次能完成的范围内（通常 5-20 分钟）
- 任务粒度参考：
  - ❌ 太粗: "实现全部后端功能"
  - ✅ 合适: "创建 TypeScript 类型定义 + API 客户端"
  - ❌ 太细: "创建 User 类型中的 name 字段"

### 0.4 确认后开工

向用户展示任务清单，确认：
- 优先级是否正确
- 是否有遗漏或需要排除的
- 技术栈确认

确认后开始执行。

## Phase 1: 项目初始化

**目标**: 项目可运行，Git 可追溯。

### 步骤

1. 创建项目目录 + `git init`
2. 初始化脚手架（如 `create-tauri-app`、`npm init` 等）
3. 创建 `.gitignore`（忽略 node_modules/target/dist 等）
4. **首次 commit**: `init: project scaffold`
5. 本地验证：`npm install` / `cargo build` 等能通过

### 注意

- 如果用 coding agent，**只给一个任务**，不要一次性给所有任务
- coding agent prompt 中明确说"只做项目初始化，不要做业务代码"

---

## Phase 2~N: 逐任务执行

**核心循环**，每个任务重复：

### Step 1: 准备 prompt

给 coding agent 的 prompt 必须包含：
- **明确的目标**：一句话说清楚要做什么
- **上下文**：项目路径、已完成的功能、依赖的文件
- **约束**：不要做什么、必须遵守什么
- **Done 标准**：如何判断完成

### Step 2: 执行 + 监控

- 用 `exec background:true` 启动 coding agent
- 记录 sessionId
- 如果超过预期时间没动静，检查进度

### Step 3: 验证

coding agent 完成后：
1. 检查生成的文件是否完整
2. 运行构建/编译确认无错误
3. 如果有问题，重新给 coding agent 修（保留上下文）

### Step 4: Git commit + 标记

```bash
git add -A && git commit -m "feat: {task description}"
```

### Step 5: 更新任务状态

- 本地 tasks.md：⬜ → ✅，填入 commit hash
- 飞书子任务：标记完成

### Step 6: 通知用户

每个任务完成后通知：
```
✅ 任务 {N}/{M} 完成: {任务名}
{简要说明做了什么}
下一个: {下一个任务名}
```

用户可以在此时：
- 调整后续任务
- 要求修改当前任务
- 暂停/继续

---

## Phase Last: 集成验证 + 最终总结

### 验证清单

1. `npm install && npm run build` 通过（或等价命令）
2. 核心功能手动验证（或描述验证步骤）
3. 无遗留的 TODO/FIXME（或明确记录）
4. Git 历史干净（每个 commit 对应一个任务）

### 最终总结

更新 `research/{project-name}-tasks.md`：
- 全部任务最终状态
- 总耗时
- 遇到的问题和解决方案
- 后续 TODO（如果有）

更新 MEMORY.md — 新增项目记录。

标记飞书总任务为**已完成**，通知用户。

## Coding Agent 使用规范

### 何时用 coding agent

- **用**: 新建文件 >3 个、需要探索目录结构、需要反复试错
- **不用**: 改一个文件几行、创建单个配置文件、写单个脚本

### Prompt 三要素

每次给 coding agent 的 prompt 必须包含：

```
## 任务：{一句话目标}

### 已完成
- 列出已完成的任务和关键文件路径

### 你需要做的
- 具体要创建/修改哪些文件
- 关键代码逻辑（如有参考给出代码片段）

### 约束
- 不要修改的文件
- 必须遵守的模式/规范

### Done 标准
- 可检查的完成条件
```

### 防止 coding agent "摸鱼"

- **不要给开放式任务**如"实现用户系统"，而是"创建 `src/api/auth.ts`，包含 login/logout 两个函数"
- **不要一次给太多任务**，一个任务一个 agent run
- **检查输出**：agent 说"完成"不代表真的完成，必须验证文件存在且内容正确
- **给具体文件路径**，不要让 agent 自己决定放哪里

### 通知机制

每个 coding agent 任务 prompt 末尾加上：
```
When completely finished, run:
openclaw system event --text "Done: {brief summary}" --mode now
```

## Git 规范

### Commit 格式

```
{type}: {description}

类型:
- init: 项目初始化
- feat: 新功能
- fix: 修复
- refactor: 重构
- chore: 构建/配置/依赖
- docs: 文档
```

### 分支策略

- 简单项目：直接 main 分支
- 复杂项目：feature/{name} → 合并回 main

### 关键规则

- 每个任务完成后必须 commit（保证可追溯）
- 不要在 commit 中混合多个任务
- `.env` / token 等敏感文件不进 git

## 飞书任务同步

### 同步时机

| 事件 | 动作 |
|------|------|
| Phase 0 完成 | 创建总任务 + 所有子任务 |
| 单个任务完成 | 标记子任务完成 |
| 任务失败/阻塞 | 更新子任务描述，说明原因 |
| 全部完成 | 标记总任务完成 |

### 子任务格式

- 标题: `{序号}. {任务名}`
- 描述: Done 标准 + 关键产出
- 截止时间: 按预估时间设置

### 注意

- 飞书任务是用户的**可视仪表盘**，必须实时更新
- 如果飞书 API 调用失败，不影响本地开发，记录错误即可
- 用户可以通过飞书随时查看进度，不需要来问

---

## 中断恢复

如果中途被中断（context 换了、session 重启）：

1. 读 `research/{project-name}-tasks.md` 恢复上下文
2. 检查 git log 确认最后一个 commit 对应哪个任务
3. 从下一个未完成的任务继续
4. 通知用户 "恢复开发，继续任务 {N}"

## 与 learning-workflow 的区别

| 维度 | dev-workflow | learning-workflow |
|------|-------------|-------------------|
| 目标 | 产出可运行的项目/功能 | 学习知识 + 沉淀文档 |
| 跟踪 | Git commit + 任务清单 | 飞书任务 + 学习笔记 |
| 产出物 | 代码 + 可运行程序 | 研究文档 + 知识库 |
| 验证 | 构建/运行通过 | 文档完整性 + 知识库同步 |
| coding agent | 每个任务一个 run | 可批量并行 |
