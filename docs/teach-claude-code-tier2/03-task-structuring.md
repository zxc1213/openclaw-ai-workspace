## 问题：为什么"随便说说"效果差

先看几组对比：

| 自由格式 | 问题 |
|---------|------|
| "帮我优化一下代码" | 优化什么？性能？可读性？体积？范围多大？ |
| "加个缓存" | 缓存什么？用什么方案？缓存过期策略？ |
| "修一下那个 bug" | 哪个 bug？在哪个文件？什么条件下触发？ |

Claude Code 不是读心术师。你给它的信息越模糊，它越需要"猜"——猜错了就要返工，浪费你的时间和它的 token。

**核心原则：输入质量决定输出质量。** 花 30 秒组织好任务描述，能省下 10 分钟的返工。

## 四要素模板

```
## 任务：[一句话标识]

### Goal
[一句话说清楚要做什么、产出什么]

### Context
- [关键背景信息，不超过 5 行]
- [相关文件路径]

### Constraints
1. [不能做什么 / 必须遵守什么]

### Done
1. [怎么算完成了]
```

### 四要素详解

#### Goal（目标）—— 一句话，能复述就算理解了

```
❌ "帮我搞一下数据库连接"
✅ "将 SQLite 连接切换为 PostgreSQL，使用 connection pool，保持现有 API 行为不变"
```

Goal 的标准：**如果你让另一个人复述这个任务，他能说清楚该做什么。**

#### Context（背景）—— 只给必要信息

```
### Context
- 项目使用 Prisma ORM，schema 在 prisma/schema.prisma
- 当前连接配置在 src/config/database.ts
- 现有测试在 tests/ 目录，使用 vitest
- 参考 Prisma PostgreSQL 文档：https://pris.ly/d/postgresql
```

Context 的标准：**删掉任何一条都不影响任务完成吗？如果是，别写。** 越少越好，但关键路径必须有。

#### Constraints（约束）—— 最多 5 条

```
### Constraints
1. 不修改 prisma/schema.prisma 中的现有模型定义
2. 使用 pg 库的 Pool，连接数上限 10
3. 数据库 URL 从环境变量 DATABASE_URL 读取
4. 所有现有测试必须通过
5. 不引入新的依赖
```

Constraints 的作用：**缩小解空间。** 约束越清晰，Claude Code 越快收敛到正确答案。

#### Done（完成标准）—— 必须可验证

```
### Done
1. src/config/database.ts 使用 PostgreSQL Pool 连接
2. pnpm test 全部通过
3. 项目能正常启动：pnpm dev
```

Done 的标准：**每一条都能用 yes/no 回答。** "代码质量提升"不是可验证的，"所有测试通过"是。

## 对比：结构化 vs 自由格式

### 场景：给 Express 项目加上速率限制

**自由格式（新手做法）：**

```
帮我给项目加上限流
```

Claude Code 的反应：
- 什么层限流？中间件级别还是路由级别？
- 用什么库？express-rate-limit？
- 限制多少请求？时间窗口多大？
- 对所有路由还是特定路由？

→ 它要么猜，要么反问你 5 个问题。

**结构化（进阶做法）：**

```markdown
## 任务：API 速率限制

### Goal
为所有 /api/ 路由添加速率限制中间件，每分钟最多 100 次请求。

### Context
- Express 4.x 项目，入口文件 src/app.ts
- 现有中间件在 src/middleware/ 目录
- 已安装 express-rate-limit（v7.x）

### Constraints
1. 只限流 /api/ 前缀的路由，静态资源不限
2. 超限时返回 429 + JSON 格式的错误信息
3. 使用 Redis 存储计数器（已有 ioredis 依赖）
4. 不修改现有路由代码，只在 app.ts 中注册中间件

### Done
1. src/middleware/rateLimiter.ts 创建完成
2. app.ts 中注册了限流中间件
3. pnpm test 全部通过
4. 用 ab 命令验证限流生效（并发 200 请求，验证部分返回 429）
```

Claude Code 的反应：
- 清楚要做什么、怎么做、在哪改
- 改完跑测试，测试不过自动修复
- 直接给出可验证的结果

→ **效率差 3-5 倍。**

## 实战：从模糊需求到精确指令

### 练习：把下面这些模糊需求改写成四要素格式

**练习 1：**
```
原文："帮我重构一下用户模块"
```

改写思路：
- Goal：具体要重构什么？（提取公共逻辑？拆分大文件？）
- Context：用户模块在哪个文件？多少行？
- Constraints：不改 API 接口？不删除功能？
- Done：怎么验证重构没破坏功能？

**练习 1 参考答案：**

```markdown
## 任务：用户模块代码拆分

### Goal
将 src/services/userService.ts（800+ 行）拆分为按职责划分的小文件，
每个文件不超过 200 行，不改变任何 API 行为。

### Context
- 当前文件：src/services/userService.ts
- 包含：认证、授权、CRUD、头像上传、偏好设置五块逻辑
- 测试：tests/services/userService.test.ts（120+ 用例）

### Constraints
1. 拆分后的文件放在 src/services/user/ 目录
2. 导出接口保持不变，其他文件不需要修改 import
3. 所有现有测试必须通过（可适当调整 import 路径）
4. 每个文件要有 JSDoc 注释说明职责

### Done
1. src/services/user/ 下有 5 个文件，每个 ≤ 200 行
2. 原 userService.ts 变为 re-export 文件
3. pnpm test 全部通过
4. pnpm lint 无错误
```

## 常见误区

### 误区 1：Context 写太多

```
❌ Context 写了 20 行项目历史、技术选型原因、团队成员信息...
   Claude Code：信息过载，关键信息被淹没

✅ Context 只写 3-5 行关键路径：文件位置、技术栈、相关依赖
```

**Claude Code 能读文件。** 你不需要在 Context 里把整个代码贴给它——告诉它在哪，它会自己读。

### 误区 2：Goal 太大

```
❌ "帮我重构整个项目"
✅ "将 userService.ts 拆分为 5 个文件"
```

大任务要拆成小任务，每次一个 Goal。

### 误区 3：没有 Done 标准

```
❌ "确保代码质量好"
✅ "pnpm test 通过 + pnpm lint 零错误 + 无 TypeScript 报错"
```

### 误区 4：Constraints 写成建议

```
❌ "如果可以的话，尽量不引入新依赖"
✅ "不引入新的 npm 依赖"
```

"如果可以的话" = 没约束。Claude Code 会判断"可以"然后引入新依赖。
