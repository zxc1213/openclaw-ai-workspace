## 什么是开发闭环

一个完整的开发闭环包含四个环节：

```
  ┌──────────┐     ┌──────────┐
  │  编写代码  │ ──→ │  运行测试  │
  └──────────┘     └──────────┘
       ↑                 │
       │                 ↓
  ┌──────────┐     ┌──────────┐
  │  修复问题  │ ←── │  检查结果  │
  └──────────┘     └──────────┘
```

**新手模式：** 写代码 → 自己跑测试 → 自己看结果 → 自己修 → 再跑测试
**进阶模式：** 描述需求 → Claude Code 写代码 → 跑测试 → 自动修复 → 迭代直到通过

区别在于：**你是监督者，不是操作者。**

## 为什么手动验证太慢

手动验证的问题：

| 环节 | 手动操作 | 耗时 |
|------|---------|------|
| 发现测试失败 | 自己跑 `npm test`，看输出 | 30 秒 |
| 定位问题 | 看错误信息，找对应代码 | 2-5 分钟 |
| 修复 | 自己改代码 | 5-10 分钟 |
| 验证修复 | 再跑测试 | 30 秒 |
| **总计一次循环** | | **8-16 分钟** |

Claude Code 自动闭环：

| 环节 | 自动操作 | 耗时 |
|------|---------|------|
| 运行测试 | `pnpm test` | 10 秒 |
| 分析失败 | 读错误输出，定位问题 | 5 秒 |
| 修复代码 | 直接改文件 | 15 秒 |
| 重新测试 | `pnpm test` | 10 秒 |
| **总计一次循环** | | **~40 秒** |

**效率差 10-20 倍。** 而且自动模式可以连续跑多轮，不需要你盯着。

## 建立自动闭环的三步

### 第一步：确保项目有可运行的测试

没有测试就没有自动验证，没有自动验证就没有闭环。

```bash
# 检查你的项目有没有测试
ls tests/
ls __tests__/
ls src/*.test.ts
ls src/*.spec.ts

# 确认测试能跑
pnpm test
# 或
npm test
```

**如果项目没有测试：**

```
你：帮我给 src/utils/dateFormat.ts 写单元测试
Claude Code：
  1. 读取 src/utils/dateFormat.ts
  2. 分析函数签名和边界情况
  3. 创建 tests/utils/dateFormat.test.ts
  4. 写入测试用例（正常 + 边界 + 错误）
  5. 运行测试确认通过
```

**先给核心模块补测试**——不需要 100% 覆盖，先覆盖最常用的 5-10 个函数。

### 第二步：让 Claude Code 跑测试（不是只改代码）

这是最关键的习惯切换：

```
❌ "帮我把 dateFormat 函数改成支持时区"
   → Claude Code 改完代码就停了，你可能忘了跑测试

✅ "帮我把 dateFormat 函数改成支持时区，改完后运行所有测试"
   → Claude Code 改完 → 跑测试 → 发现失败 → 修复 → 重跑 → 通过
```

**加一句"运行测试"**，就触发了完整闭环。

### 第三步：把测试命令写进 CLAUDE.md

确保 Claude Code 知道用哪个命令跑测试：

```markdown
## 开发命令
- pnpm test          # 运行全部测试
- pnpm test:watch    # 监听模式
- pnpm lint          # ESLint
- pnpm typecheck     # TypeScript 类型检查
```

这样你说"运行测试"，它就精确地执行 `pnpm test`，不会猜错。

## 闭环节奏控制

### 什么时候需要手动介入

不是所有事情都该全自动。以下情况需要你介入：

| 情况 | 原因 | 操作 |
|------|------|------|
| 测试连续 3 轮失败 | Claude Code 可能在错误方向上死循环 | 看看它在改什么，给个提示 |
| 改动范围超出预期 | 它可能"过度热心"地改了不相关的文件 | `git diff` 检查 |
| 需要设计决策 | 比如选择算法、确定 API 设计 | 你来决定，它来执行 |
| 涉及安全/数据 | 删数据、改权限等操作 | 必须人工确认 |

### 好的指令模板

```markdown
## 任务：给用户模块添加分页查询

### Goal
实现 GET /api/users?page=1&limit=20 分页查询接口。

### Context
- 现有用户 CRUD 在 src/services/userService.ts
- 使用 Prisma ORM，schema 见 prisma/schema.prisma
- 测试框架 vitest

### Constraints
1. 使用 Prisma 的 skip/take 实现分页
2. 返回格式：{ data: [], total: number, page: number, limit: number }
3. 默认 page=1, limit=20, limit 最大 100
4. 所有现有测试必须通过

### Done
1. 新增路由和 service 方法
2. 新增单元测试覆盖分页逻辑
3. pnpm test 全部通过
4. pnpm lint 无错误
```

注意到 Done 里写了"pnpm test 全部通过"——这就是自动闭环的触发器。

## 常见失败模式

### 失败模式 1：只改代码不验证

```
你："帮我修复那个 bug"
Claude Code：（改了代码，等待下一个指令）
你：（自己跑测试，发现没修复，回去重新说）
```

**解法：** 每个任务描述都加上"改完后运行测试确认修复"。

### 失败模式 2：测试命令不对

```
你："跑一下测试"
Claude Code：执行 `npm test`
（你的项目用 pnpm，而且测试脚本是 pnpm test:unit）
→ 命令失败，Claude Code 可能自己瞎猜别的命令
```

**解法：** 在 CLAUDE.md 里写清楚正确的测试命令。

### 失败模式 3：没有基线（测试本来就挂）

```
你："加一个新功能，跑一下测试"
Claude Code：（加功能，跑测试，3 个失败）
Claude Code：（试图修复这 3 个失败，但实际上它们是之前就存在的）
→ 浪费时间修本来就有问题的地方
```

**解法：** 让 Claude Code 改代码前，先跑一次测试确认基线是绿的：

```
"先运行 pnpm test 确认当前所有测试通过，然后再修改代码"
```

### 失败模式 4：改了不该改的文件

```
Claude Code：（为了通过测试，修改了测试本身而不是实现）
→ 测试通过了但代码没改
```

**解法：** CLAUDE.md 加红线："修复 bug 时，优先修改实现代码，不要修改测试的断言来通过测试"。

## 实战：一个完整的闭环节奏

### 场景：修复一个日期格式化 bug

```markdown
## 任务：修复日期格式化 bug

### Goal
修复 formatDate 函数在传入 null/undefined 时抛异常的 bug。

### Context
- 函数位置：src/utils/dateFormat.ts
- 现有测试：tests/utils/dateFormat.test.ts
- 复现：formatDate(null) 应返回 "—" 而不是抛异常

### Constraints
1. 不改变函数的正常输入输出行为
2. 所有现有测试必须通过
3. 新增 null/undefined/空字符串的边界测试

### Done
1. formatDate(null) 返回 "—"
2. formatDate(undefined) 返回 "—"
3. formatDate("") 返回 "—"
4. pnpm test 全部通过
```

Claude Code 的执行过程：

```
轮次 1：读取 src/utils/dateFormat.ts
轮次 2：读取 tests/utils/dateFormat.test.ts
轮次 3：修改 formatDate.ts，增加空值检查
轮次 4：在测试文件中增加 3 个边界用例
轮次 5：运行 pnpm test
轮次 6：全部通过 ✅
轮次 7：输出 git diff 供你 review
```

**全程无人工介入。你只需要最后 review 一下 diff。**

## 进阶：TDD 驱动的闭环

更高级的用法是 **TDD（测试驱动开发）**：

```
1. 先写测试（描述期望行为）
2. 运行测试（确认失败 = Red）
3. 写实现代码
4. 运行测试（确认通过 = Green）
5. 重构优化
```

在 Claude Code 中这样用：

```
"用 TDD 方式给 formatDate 函数添加相对时间功能（如 '3小时前'）：
1. 先写测试描述期望行为
2. 跑测试确认失败
3. 再写实现代码
4. 跑测试确认通过
5. 最后重构优化"
```

TDD 的好处：
- **先想清楚要什么**（写测试就是写需求文档）
- **有安全网**（改实现不怕改坏，测试会告诉你）
- **不写多余的代码**（只写让测试通过的最小实现）

## 立即行动

### 今天的练习

1. **检查你的项目**：有测试吗？测试能跑吗？
2. **如果没有测试**：让 Claude Code 给 3 个核心函数写测试
3. **找一个真实 bug**，用四要素模板 + 闭环指令修复它
4. **观察**：Claude Code 自动跑了多少轮？哪里需要你介入？
