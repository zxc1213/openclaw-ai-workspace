## 什么是 Skill

Skill 是一段**预写好的操作指令**，告诉 Claude Code "遇到这类任务时，按这个流程做"。

类比：
- **CLAUDE.md** = 项目入职手册（通用规范）
- **Skill** = 岗位 SOP（具体操作流程）

```markdown
# 没有 Skill 时，每次都要说：
"帮我做代码审查。先看改动文件列表，然后逐个检查：
1. 有没有安全问题（SQL注入、XSS）
2. 有没有性能问题（N+1 查询、大循环）
3. 代码风格是否一致
4. 测试是否覆盖了改动
最后给出 approve / request changes 结论"

# 有 Skill 后，只需说：
"帮我审查这个 PR"
```

**Skill 的价值：** 把"每次 50 字的指令"变成"一句话触发"。

## 什么时候该做 Skill

**黄金法则：做过 5 次 + 未来还会做 10 次 → 值得做 Skill。**

具体信号：

| 信号 | 例子 |
|------|------|
| 你反复粘贴同一段 prompt | "每次都要告诉它先跑测试再提交" |
| Claude Code 经常忘了某个步骤 | "它总是忘了检查类型错误" |
| 流程有明确的步骤顺序 | "代码审查有固定检查清单" |
| 跨项目通用 | "Git 工作流、部署流程、测试策略" |

**什么时候不该做：**

- 只做一次的一次性任务
- 没有固定步骤的探索性任务
- 流程还在频繁变化（先稳定再做）

## Skill 的三组件

一个 Skill 由三部分组成：

### 1. 名称（简短、动词开头）

```
✅ code-review        （代码审查）
✅ deploy-check       （部署前检查）
✅ bug-fix-workflow   （Bug 修复流程）

❌ 一些关于代码审查的想法
❌ 重要提醒
❌ 技巧集合
```

### 2. 描述（100 字以内，决定触发率）

描述是最容易被忽视但最重要的部分。Claude Code 通过描述来判断"这个任务该不该用这个 Skill"。

### 3. 指令（具体的操作步骤）

这是 Skill 的主体——告诉 Claude Code 按什么步骤执行。

---

## 描述四要素：决定触发率的关键

好的描述包含四个要素：

```
┌─────────────────────────────────────────────┐
│           描述四要素                          │
│                                              │
│  1. 能力：这个 Skill 能做什么？               │
│  2. 触发条件：什么时候应该用它？               │
│  3. 上下文：在什么场景下有效？                 │
│  4. 边界：什么时候不该用它？                   │
│                                              │
└─────────────────────────────────────────────┘
```

### 对比示例

**❌ 低效描述（只写能力）：**

```markdown
代码审查工具。检查代码质量、安全性和风格。
```

问题：Claude Code 不知道什么时候该用、什么时候不该用。可能在你只想"解释一下这段代码"时也触发它。

**✅ 高效描述（四要素齐全）：**

```markdown
对 Git diff 或指定文件进行代码审查。检查安全性（注入、XSS）、
性能（N+1、内存泄漏）、代码风格一致性，给出 approve/changes 结论。

触发：code review、审查代码、PR review、检查代码。
适用：Git diff、单个文件、指定目录。
不适用：架构级评审（用 architecture-review）、解释代码（直接问）。
```

**对比效果：**
- 低效描述：触发时经常"文不对题"
- 高效描述：几乎每次触发都恰到好处

## 菜单式设计：避免 Skill 臃肿

### 问题：一个 Skill 塞太多内容

新手常犯的错误：把所有相关的东西都塞进一个 Skill，越写越大。

```markdown
# ❌ 臃肿的 Skill（500+ 行）
## 代码审查
### 安全检查
（50 行详细的安全检查规则）
### 性能检查
（50 行详细的性能检查规则）
### 风格检查
（50 行详细的风格检查规则）
### 测试检查
（50 行详细的测试检查规则）
### 架构检查
（50 行详细的架构检查规则）
### 文档检查
（50 行详细的文档检查规则）
```

问题：每次触发都加载 500 行，大部分内容用不上，浪费上下文窗口。

### 方案：菜单式设计

```markdown
# ✅ 菜单式 Skill（主文件 20 行）

## 代码审查
对代码变更进行系统性审查。

### 审查维度（根据需要选择）：
1. **安全审查** → 读取 references/security-checklist.md
2. **性能审查** → 读取 references/perf-checklist.md
3. **风格审查** → 读取 references/style-checklist.md
4. **测试审查** → 读取 references/test-checklist.md

默认执行全部维度。如果用户指定了某个维度，只执行对应的检查。
```

好处：
- **主文件只加载 20 行**——不浪费上下文
- **按需深入**——需要什么读什么
- **易于维护**——改安全规则不影响性能规则

### 文件结构

```
skills/code-review/
  SKILL.md                    # 菜单（20 行）
  references/
    security-checklist.md     # 安全检查详细规则
    perf-checklist.md         # 性能检查详细规则
    style-checklist.md        # 风格检查详细规则
    test-checklist.md         # 测试检查详细规则
```

## 一个完整的 Skill 实例

以"代码审查"为例，展示一个生产可用的 Skill：

```markdown
# 代码审查 (code-review)

对 Git diff 或指定文件进行系统性审查，给出结构化报告。

## 触发条件
用户提到"审查代码"、"code review"、"PR review"、"检查这段代码"。

## 使用方法
1. 确认审查范围（全部 diff / 指定文件 / 指定目录）
2. 读取 references/ 对应维度的检查清单
3. 逐维度检查，记录发现的问题
4. 输出审查报告

## 审查维度
| 维度 | 文件 | 检查内容 |
|------|------|---------|
| 安全 | security-checklist.md | 注入、XSS、认证绕过 |
| 性能 | perf-checklist.md | N+1、内存泄漏、大循环 |
| 风格 | style-checklist.md | 命名、结构、一致性 |
| 测试 | test-checklist.md | 覆盖率、边界用例 |

## 输出格式
```markdown
## 审查报告

### ✅ 通过项
- [x] ...

### ⚠️ 建议改进
- [ ] ...（文件:行号，建议：...）

### ❌ 必须修改
- [ ] ...（文件:行号，问题：...，建议：...）

### 结论
APPROVE / REQUEST CHANGES
```

## 不适用场景
- 解释代码逻辑（直接问，不需要审查流程）
- 架构级评审（范围太大，先单独讨论）
- 非代码文件（文档、配置文件等）
```

## 渐进式加载：先定位再深入

Skill 的加载分三个阶段，像一个**从目录到详情的查找过程**：

```
阶段 1：元信息（~100 tokens）
  └─ Agent 看到描述，判断"要不要用这个 Skill"

阶段 2：完整指令（≤ 5000 tokens）
  └─ Agent 读取 SKILL.md，了解"怎么做"
  └─ 发现需要详细检查清单 → 触发阶段 3

阶段 3：资源文件（按需）
  └─ Agent 读取 references/ 下的具体文件
  └─ 获得完整的检查规则
```

**为什么要这样设计？**

因为上下文窗口是有限资源。如果你有 20 个 Skill，每个 5000 tokens，全部加载就要 100K tokens——直接爆了。

渐进式加载意味着：
- 20 个 Skill 的描述（元信息）= ~2000 tokens（很小）
- 通常只有 1-2 个 Skill 被完整加载 = ~10000 tokens（可控）
- 资源文件按需加载 = 几乎不增加开销
