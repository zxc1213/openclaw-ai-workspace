## 什么是 Harness

OpenAI 在 2026 年初提出了一个公式：

> **Agent = Model × Harness**

- **Model**：Claude/GPT/GLM 的推理能力（引擎）
- **Harness**：配置文件 + 工具 + 记忆 + 流程（方向盘和仪表盘）

打个比方：
- 没有 Harness 的 Claude，像一个**蒙着眼跑步的人**——有力气但没方向
- 有 Harness 的 Claude，像一个**有 GPS、有地图、有导航的司机**——知道往哪走、走哪条路

**Harness 不改模型，但改变模型输出质量。** 这是提升 Agent 效果性价比最高的方式。

## 同一模型，不同 Harness，天差地别

### 真实数据

| 场景 | Harness 级别 | 效果 |
|------|-------------|------|
| 裸 Claude API | 无（只有 system prompt） | 能聊天，不能操作 |
| Claude Code 基础用法 | 内置工具，无项目配置 | 能改文件、跑命令，但经常搞错项目约定 |
| Claude Code + CLAUDE.md | 有项目规范 | 理解项目结构，按规范操作 |
| Claude Code + CLAUDE.md + Skills + MCP | 完整 Harness | 能处理复杂工作流，少犯错，效率高 |

**最震撼的数据来自 OpenAI 官方**：LangChain 团队用同一模型，只改 Harness 配置，任务成功率从 52.8% 提升到 66.5%。没换模型，只改了"怎么用模型"。

### 一个生活中的类比

```
裸 Claude：       "给我一把锤子"（你有工具，但不知道钉哪）
+ CLAUDE.md：     "图纸在这里"（你知道该做什么了）
+ Skills：        "这是标准工序"（你知道怎么做最高效）
+ MCP：           "材料库在这"（你能接触到更多资源）
```

每加一层 Harness，效率就上一个台阶。

### 组件全景

```
│                   Your Harness                    │
│                                                  │
│  ┌──────────┐  ┌──────────┐  ┌───────────────┐  │
│  │CLAUDE.md │  │  Skills  │  │     MCP       │  │
│  │ 项目规范  │  │ 可复用流程│  │  外部工具接入  │  │
│  └──────────┘  └──────────┘  └───────────────┘  │
│  ┌──────────┐  ┌──────────┐  ┌───────────────┐  │
│  │ 记忆系统  │  │  Hooks   │  │  Subagents    │  │
│  │ 文件/git │  │  事件钩子 │  │  子任务分发    │  │
│  └──────────┘  └──────────┘  └───────────────┘  │
│                                                  │
│  ─────────────── Claude Code ──────────────────  │
│  ─────────────── Shell + Editor ───────────────  │
│  ─────────────── Anthropic Model ──────────────  │
└─────────────────────────────────────────────────┘
```

### 各组件的作用

| 组件 | 解决什么问题 | 优先级 |
|------|------------|--------|
| **CLAUDE.md** | Agent 不知道项目约定、技术栈、代码风格 | ⭐⭐⭐ 必做 |
| **Hooks** | 每次操作前/后自动执行检查 | ⭐⭐ 推荐 |
| **Skills** | 反复用的复杂流程每次都要重新描述 | ⭐⭐ 推荐 |
| **MCP** | Agent 访问不到外部服务（数据库、API） | ⭐ 进阶 |
| **Subagents** | 复杂任务需要并行或隔离执行 | ⭐ 进阶 |
| **记忆系统** | 新对话丢失之前的关键决策 | ⭐⭐ 推荐 |

**新手的第一步：写好 CLAUDE.md。** 这一个动作就能带来最明显的效果提升。

## 从裸用到配置化：三个必做项

### 必做项 1：项目根目录放 CLAUDE.md

这是 Claude Code 每次启动时自动读取的配置文件。相当于给新加入项目的队友一份入职手册。

**最小可用版 CLAUDE.md：**

```markdown
# 项目说明

- 技术栈：Node.js + TypeScript + Express
- 包管理器：pnpm
- 测试框架：vitest
- 代码规范：ESLint + Prettier

## 项目结构

src/
  routes/     # API 路由
  services/   # 业务逻辑
  models/     # 数据模型
  middleware/  # 中间件
tests/        # 测试文件

## 开发命令

- 启动：pnpm dev
- 测试：pnpm test
- 检查：pnpm lint

## 代码规范

- 使用 async/await，不用 .then()
- 错误处理统一用 try/catch
- 新文件必须写测试
- TypeScript strict 模式开启

## 安全红线

- 不要提交 .env 文件
- 不要修改 database/migrations/ 下的已有文件
- SQL 查询必须用参数化，禁止字符串拼接
```

**为什么这能立竿见影？**

因为 Claude Code 现在知道：
- 用什么命令跑测试（不会猜成 `npm test` 然后失败）
- 代码风格是什么（不会用 `.then()` 写你想要 `await` 的代码）
- 哪些东西不能碰（不会乱改数据库迁移文件）

### 必做项 2：养成"先 commit 再改"的习惯

```bash
# 每次让 Claude Code 改代码前：
git add -A && git commit -m "before: 让 Claude Code 改 XXX"

# 改完后检查：
git diff    # 看看它到底改了什么
git reset   # 不满意？一键回滚
```

这比任何配置都有效——它是你的安全网。

### 必做项 3：让它跑测试，不要只改代码

```
❌ "帮我加上用户认证"
✅ "帮我加上用户认证，改完后运行测试确保通过"
```

加上"跑测试"这一句，就触发了 Agentic Loop 的验证环节，出错会自动修复。

## Harness 投入的 ROI

| 投入 | 时间 | 收益 |
|------|------|------|
| 写一个基础 CLAUDE.md | 15 分钟 | 新对话不用重复解释项目，节省每次 5-10 分钟 |
| 配置 Hooks 自动跑 lint | 30 分钟 | 每次 commit 前自动检查，减少低级错误 |
| 做一个 Skill（如代码审查） | 1-2 小时 | 以后每次审查不用重新描述标准 |
| 接入 MCP | 2-4 小时 | Agent 能直接查数据库/调 API，不用你手动搬运数据 |

**投入产出比最高的永远是 CLAUDE.md。** 15 分钟写好，每次新对话都受益。

## 立即行动

### 今天的练习

1. **打开你正在用的项目**，检查根目录有没有 CLAUDE.md
2. **如果没有**，花 15 分钟写一个最小可用版（参考上面的模板）
3. **如果有**，读一遍，问自己：它足够精确吗？有没有遗漏的约定？
4. **测试效果**：开一个新的 Claude Code session，让它做一个任务，对比有没有 CLAUDE.md 的差异

### CLAUDE.md 质量检查清单

- [ ] 技术栈和版本写清楚了吗？（"Node.js" 还是 "Node.js 20 + TypeScript 5.3"）
- [ ] 常用命令列全了吗？（dev/test/build/lint/deploy）
- [ ] 项目结构有说明吗？（哪个目录放什么）
- [ ] 代码规范有 3-5 条核心规则吗？（不用写 50 条，5 条就够）
- [ ] 安全红线标了吗？（哪些目录/文件不能动）
- [ ] 总长度在 200 行以内吗？（太长反而会稀释关键信息）
