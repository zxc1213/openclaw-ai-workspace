## 一个例子：两种用法的天壤之别

**方式 A——当搜索引擎用：**

```
你：帮我写一个 Python 函数，反转链表
Claude：好的，这是反转链表的代码...
（你复制粘贴，手动调试，自己集成到项目）
```

**方式 B——当编程搭档用：**

```
你：帮我把 src/linked_list.py 里的反转链表函数改成迭代版本，
    同时更新对应的单元测试

Claude Code：
  1. 读取 src/linked_list.py，理解当前递归实现
  2. 读取 tests/test_linked_list.py，了解现有测试结构
  3. 修改代码为迭代版本
  4. 运行测试，确认全部通过
  5. 如果测试失败，自动修复并重跑

你：（review 一下改动，满意的话 git commit）
```

区别在哪？方式 B 里 Claude Code **自己读文件、自己改代码、自己跑测试**。你只需要描述你要什么，它自己想办法完成。这就是 **Agentic Coding**。

## Agentic Loop：Agent 的核心引擎

Claude Code 的底层运行机制叫 **Agentic Loop**（智能体循环）：

```
┌─────────────────────────────────────────┐
│          Agentic Loop                    │
│                                          │
│   1. 获取上下文 ──→ 读取文件、理解需求    │
│          ↓                                │
│   2. 决策规划   ──→ 想清楚怎么做          │
│          ↓                                │
│   3. 执行操作   ──→ 调用工具（改文件/跑命令）│
│          ↓                                │
│   4. 验证结果   ──→ 跑测试、检查输出      │
│          ↓                                │
│   需要修复？──→ 回到步骤 1               │
│   完成了？  ──→ 等待下一个指令            │
└─────────────────────────────────────────┘
```

关键点：**它不是"你说一句它回一句"，而是"你说一个目标，它自己循环执行直到完成"。**

### 一个真实循环的例子

假设你让 Claude Code "给这个 Express 项目加上用户认证"：

| 轮次 | 动作 | 说明 |
|------|------|------|
| 1 | 读取 `package.json`、`app.js` | 理解项目结构 |
| 2 | 读取 `routes/` 目录 | 了解现有路由 |
| 3 | 修改 `app.js` 加入 JWT 中间件 | 代码变更 |
| 4 | 创建 `middleware/auth.js` | 新文件 |
| 5 | 运行 `npm install jsonwebtoken bcrypt` | 安装依赖 |
| 6 | 创建 `routes/auth.js`（登录/注册） | 新路由 |
| 7 | 运行 `npm test` | 验证 |
| 8 | 测试失败（缺少 mock） | 发现问题 |
| 9 | 修复测试 mock | 自我修复 |
| 10 | 运行 `npm test`，全部通过 | 完成 |

**整个过程你只说了一句话，它自己循环了 10 轮。** 这就是 Agentic Loop 的威力。

## 工具体系：Agent 的"手和眼"

Claude Code 能做的事，取决于它能用的工具：

### 核心工具清单

| 工具 | 能力 | 类比 |
|------|------|------|
| 文件读写 | 读取、创建、修改文件 | ✏️ 手 |
| Shell 执行 | 运行任意终端命令 | 🖥️ 手 |
| 搜索 | 在代码库中搜索 | 🔍 眼 |
| 浏览器 | 访问网页、查文档 | 🌐 眼 |
| Git | commit、branch、diff | 📦 手 |
| MCP | 连接外部服务 | 🔌 手 |

### 为什么 Shell Access 是杀手级能力

普通聊天机器人只能"告诉你"该做什么。Claude Code 能直接执行命令：

```bash
# Claude Code 可以自己运行这些：
npm test
python -m pytest
git diff
docker compose up
cargo build
```

这意味着它能**立即验证自己的改动是否正确**。写完代码→跑测试→发现错误→修复→再跑测试，整个循环全自动。

### 工具的安全边界

但强大的工具也需要安全约束：

```
❌ Claude Code 不该做的事：
   - 删除生产数据库
   - git push --force 到共享分支
   - 部署到线上环境
   - 修改 ~/.ssh/config

✅ 安全的默认操作：
   - 读写项目目录内的文件
   - 运行测试、lint、build
   - git commit 到本地分支
   - 安装项目依赖
```

在 CLAUDE.md 中写清楚安全红线，就是告诉 Claude Code "哪些事绝对不能做"。

## 从一次性到持续协作者：心智切换

### ❌ 新手心态（一次性问答）

- 把 Claude Code 当高级 ChatGPT
- 每次新对话都从零开始解释项目背景
- 复制代码自己集成

### ✅ 进阶心态（持续协作）

- 把 Claude Code 当队友，给它项目和规范
- 用 CLAUDE.md 把项目知识写下来，新对话自动加载
- 让它自己完成改码→测试→提交的完整流程

### 一个关键认知

> **"Agent 看不到的信息，等于不存在。"**

如果你不把项目约定写在 CLAUDE.md 里，Claude Code 就不会知道。它不会"猜"你的代码风格、目录结构、技术选型——你需要明确告诉它。

## 成熟度路线：你现在在哪？

| 级别 | 使用方式 | 典型场景 |
|------|---------|---------|
| **L0** | 偶尔问一句"帮我写个函数" | 效率和复制粘贴差不多 |
| **L1** | 让它直接修改文件、跑测试 | 比自己写快 2-3 倍 |
| **L2** | 写了 CLAUDE.md，配置了项目规范 | 新对话不用重复解释 |
| **L3** | 有 Skills、MCP、自动化流程 | Agent 能处理复杂工作流 |

**大多数新手卡在 L0→L1**。本系列后面的文章就是帮你突破这个阶段。

## 立即行动

### 今天的练习

1. 打开一个真实项目，启动 Claude Code
2. 试试这个指令（不要用示例项目，用你自己的）：
   ```
   帮我阅读整个项目结构，然后总结：
   - 技术栈是什么
   - 入口文件在哪里
   - 有哪些核心模块
   - 测试覆盖率如何
   ```
3. 根据它的总结，问它："帮我给这个项目写一个 CLAUDE.md"
4. 试试让它执行一个完整的任务：修一个 bug 或加一个小功能

### 避坑提示

- ⚠️ 第一次用 Claude Code 改文件前，确保 git 仓库是干净的（已 commit 或 stash）
- ⚠️ 让它跑命令时，先看看它要跑什么——养成 review 习惯
- ⚠️ 别让它删除重要文件，养成 `git diff` 后再确认的习惯
