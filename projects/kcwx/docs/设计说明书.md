## 1 引言

### 1.1 编写目的

本软件设计说明书详细描述了河南交投集团科技创新服务平台（以下简称"本平台"）的总体设计、架构方案、模块划分、数据库设计、接口规范、安全机制及部署方案。本文档面向技术开发人员、系统架构师和项目管理人员，旨在为系统的开发、测试、部署和维护提供技术依据。

### 1.2 项目背景

河南交通投资集团有限公司作为省属大型国有企业，下属单位众多，科技管理工作涉及项目申报、成果管理、专家评审等多个环节。传统的手工管理模式效率低下、信息孤岛严重、统计口径不统一。为提升科技管理信息化水平，集团决定建设统一的科技创新服务平台，实现科技管理工作的数字化和规范化。

## 2 设计概述

### 2.1 设计原则

本平台的设计遵循以下核心原则：

1. **高内聚低耦合：** 各微服务模块职责单一，服务间通过标准接口通信，降低模块间依赖。
2. **可扩展性：** 采用微服务架构，各服务可独立扩展，支持水平扩展应对高并发场景。
3. **安全性：** 采用国密SM2加密算法保障数据传输安全，OAuth2协议实现统一认证授权。
4. **易维护性：** 统一的代码规范、日志规范和监控体系，便于日常运维和故障排查。
5. **高可用性：** 关键服务支持集群部署，通过熔断降级机制保障系统整体可用性。
6. **标准化：** 遵循行业标准和集团信息化规范，确保与现有系统的兼容性。

### 2.2 设计目标

1. **业务目标：** 覆盖集团科技管理全业务流程，实现项目、成果、专家等核心业务的信息化管理。
2. **性能目标：** 核心接口响应时间不超过500ms（95%分位），支持500并发用户同时在线操作。
3. **可用性目标：** 系统可用性不低于99.5%，年故障时间不超过43.8小时。
4. **安全目标：** 通过等保二级认证，实现数据传输加密、访问控制、操作审计。
5. **兼容性目标：** 支持主流浏览器和移动端访问，与集团现有OA系统实现单点登录对接。

## 3 系统架构设计

### 3.1 总体架构

本平台采用基于Spring Cloud的微服务架构，整体架构分为四层：

**第一层：接入层（Access Layer）**

- Nginx反向代理服务器，负责负载均衡、SSL终止、静态资源服务
- 统一入口，支持HTTPS协议

**第二层：网关层（Gateway Layer）**

- Spring Cloud Gateway作为API网关
- 统一处理路由转发、身份认证、权限校验、限流熔断、日志记录
- 将外部请求分发至对应的后端微服务

**第三层：服务层（Service Layer）**

由12个微服务模块组成，各服务独立部署、独立数据库：

| 序号 | 服务模块 | 功能说明 |
|------|----------|----------|
| 1 | eureka-service | 服务注册中心，提供服务注册与发现 |
| 2 | gateway-service | API网关，统一入口和路由管理 |
| 3 | common-service | 公共基础服务，提供通用工具和基础组件 |
| 4 | user-oauth-service | 用户认证授权服务，处理登录、鉴权、令牌管理 |
| 5 | project-service | 项目管理服务，处理项目全生命周期业务逻辑 |
| 6 | achievement-service | 成果管理服务，处理8类科技成果的登记审核 |
| 7 | expert-service | 专家管理服务，处理专家库维护和抽取 |
| 8 | statistics-service | 统计分析服务，处理数据汇总和简报生成 |
| 9 | meeting-service | 会议管理服务，处理会议安排和纪要 |
| 10 | community-service | 社区交流服务，处理帖子发布和互动 |
| 11 | promotion-service | 成果推广服务，处理成果展示和推广申请 |
| 12 | market-service | 科技超市服务，处理需求和供给对接 |

**第四层：数据层（Data Layer）**

- MySQL 8.0：各微服务独立数据库，实现数据隔离
- Redis：分布式缓存，存储会话信息、热点数据、验证码等
- RabbitMQ：异步消息处理，用于通知推送、邮件发送、数据同步等场景

### 3.2 服务治理

#### 3.2.1 服务注册与发现

采用Eureka作为服务注册中心，各微服务启动时自动注册到Eureka Server。Eureka Client定期向Server发送心跳以维持注册状态。服务间调用通过Eureka获取目标服务实例列表，实现客户端负载均衡。

#### 3.2.2 服务间通信

服务间通信采用两种方式：
- **同步调用：** 使用OpenFeign声明式HTTP客户端进行服务间调用
- **异步消息：** 使用RabbitMQ进行解耦场景下的异步通信，如审批通知、数据同步等

#### 3.2.3 熔断与降级

引入Hystrix实现熔断降级机制。当某服务出现故障或响应超时时，自动触发熔断，返回兜底数据或友好提示，防止故障级联扩散。

#### 3.2.4 配置管理

采用Spring Cloud Config实现集中配置管理，各微服务的配置信息统一存储在Git仓库或配置服务器中，支持配置的动态刷新。

## 4 功能模块设计

### 4.1 用户认证模块（user-oauth-service）

#### 4.1.1 功能概述

统一管理用户身份认证和授权，支持三种登录方式，基于OAuth2协议实现令牌管理。

#### 4.1.2 核心组件

- **AuthenticationController：** 登录接口入口，处理账号密码登录、短信验证码登录、微信OAuth2登录
- **TokenService：** JWT令牌生成、验证和刷新，支持Access Token和Refresh Token双令牌机制
- **SmsService：** 短信验证码生成、发送和验证，集成短信服务商API
- **WechatService：** 微信OAuth2授权对接，处理微信登录的授权码换取和用户信息获取
- **SM2CryptoService：** 基于SM2国密算法的加解密服务，用于敏感数据（如密码）的加密存储和传输加密

#### 4.1.3 登录流程

**账号密码登录流程：**
1. 前端提交用户名和SM2加密后的密码
2. 网关层进行基本参数校验和限流
3. 认证服务验证用户名密码
4. 验证通过后生成JWT Access Token（有效期2小时）和Refresh Token（有效期7天）
5. 将用户会话信息写入Redis缓存
6. 返回令牌信息给前端，前端存储于本地

**短信验证码登录流程：**
1. 前端请求发送验证码，SmsService生成6位随机码
2. 验证码存入Redis（key: sms:code:{phone}, TTL: 5分钟）
3. 调用短信服务商API发送验证码
4. 用户提交手机号和验证码
5. 认证服务从Redis取出验证码进行比对
6. 验证通过后执行与密码登录相同的令牌生成流程

**微信登录流程：**
1. 前端请求微信授权二维码，包含回调地址和state参数
2. 用户扫码授权后，微信回调认证服务，携带授权码
3. 认证服务使用授权码向微信服务器换取Access Token和用户OpenID
4. 查询系统中已绑定的用户账号
5. 绑定存在则直接生成令牌，不存在则引导用户绑定

### 4.2 项目管理模块（project-service）

#### 4.2.1 功能概述

实现科技项目全生命周期管理，覆盖申报、审批、立项、合同、开题、执行、中期检查、验收、结题等9个阶段。

#### 4.2.2 状态机设计

项目状态流转设计为有限状态机：

```
草稿 → 待审批 → 审批中 → 已立项 → 合同签订 → 开题准备 → 执行中 → 中期检查 → 验收中 → 已验收 → 已结题
                   ↓          ↓
                 已退回      已终止
```

每个状态定义允许的转换条件和触发操作，状态转换时自动记录操作日志。

#### 4.2.3 审批流程设计

采用基于角色的多级审批机制：

1. **审批链定义：** 项目类别关联审批链，审批链定义各级审批角色和审批顺序
2. **审批任务生成：** 项目提交后，系统根据审批链自动生成各级审批任务
3. **任务分发：** 审批任务通过RabbitMQ异步通知对应审批人
4. **审批操作：** 审批人在待办列表中执行通过/退回/驳回操作
5. **流程流转：** 审批通过后自动流转至下一级，退回时退至申报人，驳回时终止流程

#### 4.2.4 核心数据实体

- **Project（项目）：** 项目基本信息、状态、所属单位、负责人等
- **ProjectMember（项目成员）：** 项目参与人员及分工
- **ProjectBudget（项目预算）：** 预算明细
- **ProjectProgress（项目进度）：** 进度报告记录
- **ApprovalTask（审批任务）：** 审批流程中的任务实例
- **ProjectContract（项目合同）：** 合同信息及附件

### 4.3 成果管理模块（achievement-service）

#### 4.3.1 功能概述

支持8类科技成果（论文、专利、软件著作权、标准、工法、科技奖励、技术规程、研究报告）的登记、审核与管理。

#### 4.3.2 设计方案

采用策略模式（Strategy Pattern）处理不同类型成果的差异：

- 定义统一的AchievementEntity基类，包含成果通用字段
- 每种成果类型继承基类，扩展特有字段
- AchievementHandler接口定义统一操作规范（创建、更新、审核）
- 各类型实现具体的AchievementHandler

#### 4.3.3 核心数据实体

- **Achievement（成果基表）：** 成果ID、类型、名称、所属项目、完成人、完成单位、状态等
- **AchievementPaper（论文扩展表）：** 期刊名称、ISSN、卷期号、影响因子等
- **AchievementPatent（专利扩展表）：** 专利号、专利类型、授权日期等
- **AchievementCopyright（软著扩展表）：** 登记号、证书编号等
- **AchievementStandard（标准扩展表）：** 标准编号、级别、发布日期等
- **AchievementMethod（工法扩展表）：** 工法级别、批准单位等
- **AchievementReward（奖励扩展表）：** 奖励名称、等级、颁发单位等
- **AchievementProcedure（规程扩展表）：** 规程编号、适用范围等
- **AchievementReport（报告扩展表）：** 报告类型、页数等

### 4.4 专家管理模块（expert-service）

#### 4.4.1 功能概述

维护集团级专家资源库，支持专家信息CRUD操作、分类管理和随机抽取功能。

#### 4.4.2 专家抽取算法

专家抽取采用基于权重约束的随机选取算法：

1. 根据评审要求设定筛选条件（专业领域、回避单位、是否可用）
2. 在符合条件的专家池中，依据专家评审次数进行权重均衡（评审次数少的权重更高）
3. 使用加权随机算法抽取候选专家名单
4. 记录抽取日志，确保抽取过程可追溯

### 4.5 统计分析模块（statistics-service）

#### 4.5.1 功能概述

提供多维度数据统计与简报生成功能，通过定时任务和按需触发两种方式产生统计数据。

#### 4.5.2 统计方案

- **数据采集：** 通过OpenFeign调用各业务微服务的数据查询接口获取原始数据
- **聚合计算：** 在内存中按统计维度进行分组聚合（Group By），支持多级维度嵌套
- **缓存策略：** 统计结果缓存至Redis，设置过期时间（日报缓存1小时，月报缓存6小时）
- **简报生成：** 使用模板引擎填充统计数据生成简报文档，支持导出PDF格式

### 4.6 会议管理模块（meeting-service）

管理科技类会议全流程，包括会议创建、通知、签到和纪要。会议通知通过RabbitMQ异步推送至参会人的消息中心。

### 4.7 社区交流模块（community-service）

提供类似论坛的社区交流功能，支持帖子发布、回复、点赞、标签和搜索。帖子内容采用全文索引（MySQL FULLTEXT）实现关键词检索。

### 4.8 成果推广模块（promotion-service）

管理创新成果的展示和推广，支持图文展示和推广审批流程。成果展示页面支持分页加载和按热度排序。

### 4.9 科技超市模块（market-service）

提供技术需求和供给的发布与对接平台。对接流程通过消息队列实现异步通知，确保供需双方及时收到对接信息。

### 4.10 公共基础模块（common-service）

提供各微服务共享的基础功能：

- **统一异常处理：** 全局异常处理器，统一异常响应格式
- **统一日志框架：** 基于Logback的结构化日志，支持日志级别动态调整
- **文件存储服务：** 统一的文件上传下载接口，支持本地存储和对象存储
- **数据字典服务：** 统一管理系统中使用的枚举值和字典数据
- **消息模板服务：** 管理通知消息模板，支持短信、站内信、邮件等多渠道

## 5 数据库设计

### 5.1 数据库总体设计

平台采用"一服务一库"的数据库隔离策略，各微服务拥有独立的MySQL数据库实例。公共基础模块维护共享的字典数据和系统配置数据，其他服务通过OpenFeign调用获取所需的基础数据。

### 5.3 数据库设计规范

1. **命名规范：** 表名使用小写下划线命名法（snake_case），字段名同样采用小写下划线
2. **主键策略：** 统一使用BIGINT自增主键，分布式场景下考虑雪花算法生成唯一ID
3. **时间字段：** 使用DATETIME类型记录创建时间（create_time）和修改时间（update_time），自动维护
4. **逻辑删除：** 采用is_deleted字段（0-未删除，1-已删除）实现软删除
5. **字符集：** 统一使用utf8mb4字符集，支持emoji和特殊字符
6. **索引策略：** 根据查询场景合理创建索引，避免过度索引影响写入性能

## 6 接口设计

### 6.1 API设计规范

#### 6.1.1 RESTful风格

平台API遵循RESTful设计规范：

- 使用HTTP方法表示操作类型：GET（查询）、POST（创建）、PUT（更新）、DELETE（删除）
- URL路径使用名词复数形式，如 `/api/v1/projects`、`/api/v1/experts`
- 使用路径参数标识资源，如 `/api/v1/projects/{id}`
- 查询参数用于过滤、排序和分页

#### 6.1.2 统一响应格式

所有API接口采用统一的JSON响应格式：

```json
{
    "code": 200,
    "message": "操作成功",
    "data": {},
    "timestamp": 1700000000000
}
```

分页查询响应格式：

```json
{
    "code": 200,
    "message": "查询成功",
    "data": {
        "records": [],
        "total": 100,
        "size": 10,
        "current": 1,
        "pages": 10
    },
    "timestamp": 1700000000000
}
```

错误响应格式：

```json
{
    "code": 40001,
    "message": "参数校验失败：项目名称不能为空",
    "data": null,
    "timestamp": 1700000000000
}
```

#### 6.1.3 接口版本管理

API路径中包含版本号前缀，如 `/api/v1/`，便于后续接口升级时保持向后兼容。

### 6.2 鉴权机制

#### 6.2.1 令牌认证流程

1. 用户登录成功后获得Access Token（JWT格式）
2. 前端在后续请求的Header中携带：`Authorization: Bearer {access_token}`
3. API网关拦截请求，解析并验证JWT令牌
4. 验证通过后将用户信息写入请求头，转发至后端微服务
5. 后端微服务从请求头获取用户信息进行业务授权判断

#### 6.2.2 令牌刷新机制

- Access Token有效期2小时，Refresh Token有效期7天
- Access Token过期后，前端使用Refresh Token换取新的Access Token
- Refresh Token过期后需重新登录

#### 6.2.3 权限控制

- **功能权限：** 基于RBAC（基于角色的访问控制）模型，用户角色关联菜单权限，网关层进行接口级权限校验
- **数据权限：** 在业务层实现数据范围控制，如单位管理员只能查看本单位数据，项目负责人只能管理自己负责的项目

### 6.3 接口签名规范

为防止接口篡改和重放攻击，对外接口采用签名机制：

1. 将请求参数按字典序排序拼接
2. 在末尾追加时间戳和AppSecret
3. 对拼接字符串进行HMAC-SHA256计算，得到签名值
4. 签名值放入请求头 `X-Signature`，时间戳放入 `X-Timestamp`
5. 服务端验证签名一致性和时间戳有效期（5分钟内）

## 7 安全设计

### 7.1 数据加密

#### 7.1.1 SM2国密加密

平台采用SM2椭圆曲线公钥密码算法进行关键数据的加密保护：

- **用户密码存储：** 用户密码使用SM2公钥加密后存储，验证时使用私钥解密比对。同时结合BCrypt算法进行哈希处理，实现双重保护。
- **敏感数据传输：** 用户登录时密码在前端使用SM2公钥加密后传输，防止中间人攻击截获明文密码。
- **密钥管理：** SM2密钥对由系统初始化时生成，私钥存储于安全配置中，支持密钥轮换。

#### 7.1.2 数据传输安全

- 全站启用HTTPS（TLS 1.2+），保证数据传输通道安全
- 敏感字段（身份证号、手机号等）在展示时进行脱敏处理

### 7.2 身份认证与授权

#### 7.2.1 OAuth2认证框架

基于Spring Security OAuth2实现统一的认证授权框架：

- **授权类型：** 支持密码模式（password）、短信验证码模式（sms_code）、微信授权码模式（wechat_code）
- **令牌存储：** JWT令牌存储于Redis，支持快速验证和主动失效
- **会话管理：** 单点登录控制，同一用户仅允许一个有效会话，后登录踢出先登录

#### 7.2.2 RBAC权限模型

- 14种预置角色覆盖不同业务场景
- 角色-菜单关联实现功能权限控制
- 角色-数据范围关联实现数据权限控制
- 支持自定义角色，灵活配置权限

### 7.3 安全防护

#### 7.3.1 API安全

- **接口限流：** 网关层基于令牌桶算法实现接口限流，防止恶意刷接口
- **SQL注入防护：** MyBatis使用`#{}`参数化查询，杜绝SQL注入风险
- **XSS防护：** 前端对用户输入进行HTML转义，后端对富文本内容进行过滤
- **CSRF防护：** 采用JWT令牌认证，天然防止CSRF攻击

#### 7.3.2 操作安全

- **操作审计：** 关键操作（登录、审批、数据修改、删除）记录操作日志，包含操作人、操作时间、操作内容、IP地址
- **防越权访问：** 所有数据操作均校验当前用户的数据权限范围
- **文件上传安全：** 校验文件类型白名单、文件大小限制，文件存储路径使用UUID重命名防止路径遍历

## 8 部署设计

### 8.1 部署架构

#### 8.1.1 生产环境部署

生产环境采用多节点集群部署方案：

- **前端部署：** Nginx集群（至少2节点），通过Keepalived实现高可用
- **网关部署：** Spring Cloud Gateway集群（至少2节点），Nginx反向代理负载均衡
- **服务注册中心：** Eureka集群（3节点，形成互相对等的集群）
- **业务微服务：** 各微服务至少部署2个实例，通过Ribbon客户端负载均衡
- **数据库：** MySQL主从复制（1主2从），Redis哨兵模式（3节点），RabbitMQ镜像队列集群
- **文件存储：** 独立文件服务器或对象存储服务

#### 8.1.2 网络架构

- 外部访问通过Nginx暴露80/443端口
- 微服务间通信通过内部网络（VLAN隔离）
- 数据库仅允许内网访问，禁止外部直连

### 8.2 部署方案

#### 8.2.1 物理部署
