## 九、Coding Agent 最佳实践 ⭐⭐⭐

**来源**：https://zhipu-ai.feishu.cn/wiki/FbOpwv5dfi6s9GkPjLFcVrN3nWd
**最后更新**：2026-03-25

### 核心要点

十大原则，从任务输入到自动化运行的完整工作流：

1. **Agent 是协作者** — 通过配置持续优化，不是一次性问答。价值来自模型能力与开发工作流的结合
2. **任务输入结构化** — Goal（目标）+ Context（上下文）+ Constraints（约束）+ Done when（完成标准）
3. **先规划再执行** — 分析需求 → 查找代码 → 确定修改模块 → 拆分步骤 → 制定实现顺序
4. **规则沉淀为项目配置** — 临时指令写在 prompt，长期规则写入项目级配置文件
5. **执行环境决定能力边界** — 三层上下文：任务上下文、项目上下文、环境上下文
6. **参与完整开发闭环** — 实现代码修改 → 编写/更新测试 → 运行测试套件 → 执行代码检查 → 审查代码变更
7. **MCP 扩展上下文** — 从"依赖人工描述"转向"依赖工具直接查询"，扩展上下文边界
8. **沉淀为 Skills** — 结构化任务模板，可复用。经验法则：反复使用的流程就做 Skill
9. **稳定流程自动化** — Skill 定义"怎么做"，Automation 决定"何时做"
10. **合理管理会话** — 独立线程、定期压缩、分支新建。会话是"工作线程"而非简单聊天记录

### 详细笔记

**成熟度路线**：任务输入 → 任务规划 → 项目规则 → 环境配置 → 执行与验证 → 工具接入 → 工作流沉淀 → 自动化运行

**任务输入结构化的四个要素详解**：
- **Goal**：明确功能或修改，如"修复 bug"、"实现接口"
- **Context**：提供代码文件、错误信息、文档、示例
- **Constraints**：代码风格、架构规则、安全要求、依赖限制
- **Done when**：测试通过、行为变化、bug 不再复现

**执行环境的关键要素**：
- 项目工作目录与代码访问权限
- 可执行命令范围（build、test、lint）
- 外部工具或数据源连接
- 团队统一的默认行为设置

**开发闭环流程图**：
```
实现代码修改 → 编写/更新测试 → 运行测试套件 → 执行代码检查 → 审查代码变更
     ↑                                                                    ↓
     └──────────────── 发现问题继续迭代 ─────────────────────────────────────┘
```

**MCP 的关键意义**：不是"多了几个工具"，而是扩展 Agent 的上下文边界，从"代码仓库内的执行者"转变为"面向真实研发环境的协作节点"。

**会话管理的策略**：
- 每个任务使用独立线程（保持上下文清晰）
- 避免线程过长（摘要/压缩）
- 分支任务创建新线程
- 定期压缩历史上下文
- 复杂场景引入多 Agent 协作

### 对 OpenClaw 的借鉴价值
- ⭐⭐⭐ **这是最核心的参考文章之一**
- **Goal-Context-Constraints-Done 框架**可以直接应用到 OpenClaw 的 task 系统和 subagent 派任务模板
- **完整开发闭环**应整合到 coding-agent skill 中
- **三层上下文**的概念（任务/项目/环境）与 OpenClaw 的 system prompt + workspace + tools 层级完美对应
- **Skills 沉淀理念**与 OpenClaw 的 SKILL.md 机制高度一致
- **会话管理策略**对 OpenClaw 的 subagent 上下文管理有直接参考价值

## 十、Coding Agent 工作原理 ⭐⭐

**来源**：https://zhipu-ai.feishu.cn/wiki/U0KnwnzjMinEz7k4ehKcoIJNnlf
**最后更新**：2026-03-25

### 核心要点

1. **Agentic Loop**：获取上下文 → 执行操作 → 验证结果（持续循环迭代）
2. **驱动双核**：模型（推理决策）+ 工具（执行操作）
3. **工具体系**：文件操作 / 搜索 / 命令执行 / Web / 代码分析
4. **扩展组件**：Skills / MCP / Hooks / Subagents（能力层）
5. **配置文件对比**：Claude Code→CLAUDE.md, Cursor→.cursor/rules, Cline→.cline/rules, Codex→AGENTS.md

### 详细笔记

**Agent 循环的三阶段**：
- 不是严格按顺序，而是交替出现
- 不同任务类型的循环模式不同：
  - 代码结构询问：只需获取上下文
  - Bug 修复：读取代码 → 修改代码 → 运行测试 → 验证结果
  - 大型重构：包含大量验证步骤

**模型的能力边界**：
- 解析多种编程语言
- 理解模块间依赖关系
- 推断实现目标所需的修改
- 复杂任务自动拆分为多步骤

**工具的五大类型**：

| 类别 | 能力 |
|------|------|
| 文件操作 | 读取、修改、创建、重命名、结构调整 |
| 搜索 | 按模式查找、正则搜索、代码库浏览 |
| 命令执行 | Shell、启动服务、运行测试、Git |
| Web | 搜索网页、获取文档、查询错误 |
| 代码分析 | 类型错误、定义跳转、引用查找（需插件）|

**运行环境对比**：

| 环境 | 代码执行位置 |
|------|-------------|
| 本地 | 用户机器 |
| 云端 | 云端虚拟机 |
| 远程控制 | 本地机器，远程界面控制 |

**安全机制**：变更回滚（快照）、权限控制（自动修改/执行命令/人工确认）

**高效使用四原则**：
1. 提供清晰任务（越具体成功率越高）
2. 提供可验证目标（测试用例或期望输出）
3. 复杂任务先规划（先分析再执行）
4. 将任务委托给 Agent（给目标与上下文，而非逐条指令）

### 对 OpenClaw 的借鉴价值
- ⭐⭐
- OpenClaw 的 exec/read/edit/browser 等工具与 Agentic Loop 的工具体系高度对应
- 配置文件的跨工具对比确认了 AGENTS.md 是社区共识的命名
- "提供清晰任务 + 可验证目标"的原则对 OpenClaw 的 subagent 任务描述有直接指导意义

## 十一、Harness 工程：构建真正让 AI 智能体有效运行的系统 ⭐⭐⭐（最震撼）

**来源**：https://zhipu-ai.feishu.cn/wiki/ZM7owzXtZiIWvBkFLnscQi5Jn5c
**最后更新**：2026-03-23

### 核心要点

OpenAI 2026.2 官方提出：**Agent = Model + Harness**

**马的隐喻**：马=AI模型（强大但不可预测），Harness=马具（约束+引导），骑手=人类工程师

**验证数据**：
- OpenAI：100w+ 行代码，零人工编写，时间约 1/10
- LangChain：同一模型仅改 Harness，52.8% → 66.5%（Terminal Bench 2.0）
- Stripe：Minions 每周 1000+ PR，Slack 发任务 → AI 全流程 → 人工审核

**三大支柱**：

| 支柱 | 核心内容 | 关键规则 |
|------|---------|---------|
| 上下文工程 | 静态（文档/规则）+ 动态（日志/CI） | 代码仓库是唯一"事实来源" |
| 架构约束 | 依赖分层、Linter、LLM 审计、结构化测试 | 限制解空间反而提升效率 |
| 熵管理 | 文档一致性 Agent、约束扫描器、模式规范 Agent | 周期性清理，防止 AI 代码库退化 |

**LangChain Harness 优化详解**：

| 变更 | 做法 | 效果 |
|------|------|------|
| 自验证循环 | 添加"完成前检查清单"中间件 | 提交前捕获错误 |
| 上下文工程 | 启动时映射目录结构 | Agent 一开始就理解代码库 |
| 循环检测 | 跟踪重复文件编辑 | 防止死循环 |
| 推理夹层 | planning/verification 高推理强度，实现中等 | 时间预算内更高质量 |

**成熟度模型**：
- **L1 个人开发者**（1-2h）：CLAUDE.md + pre-commit + 测试集 + 清晰目录
- **L2 小型团队**（1-2d）：AGENTS.md + CI 架构约束 + 共享 Prompt 模板 + 代码评审清单
- **L3 工程组织**（1-2w）：自定义中间件 + 可观测性 + 熵管理 Agent + Harness A/B 测试

**五大常见错误**：
1. 过度设计控制流（模型进步后应能轻松移除"聪明"逻辑）
2. 视为静态（需随模型一起演进）
3. 忽略文档层（最有影响力的优化往往是最简单的：更好的文档）
4. 无反馈回路（没有反馈的 Harness 是"笼子"不是"引导工具"）
5. Human-Only 文档（Agent 访问不到的知识等于不存在）

**概念层级关系**：
```
Prompt Engineering（单次交互）
    ↓
Context Engineering（模型上下文窗口）
    ↓
Harness Engineering（整个 Agent 系统）
    ↓
Agent Engineering（Agent 内部设计与任务路由）
    ↓
Platform Engineering（基础设施）
```

### 对 OpenClaw 的借鉴价值
- ⭐⭐⭐ **这是整个知识库中最重要的文章之一**
- **核心洞察**："相同模型，不同 Harness，结果天差地别" — OpenClaw 已有不错的模型配置，提升空间在 Harness 层
- **三支柱**直接对应 OpenClaw 的优化方向：
  - 上下文工程 → AGENTS.md/SOUL.md/MEMORY.md 精确化
  - 架构约束 → workspace 目录规范 + lint + 安全机制
  - 熵管理 → heartbeat 中的文档一致性检查 + 代码质量扫描
- **成熟度模型**可以作为 OpenClaw 个人/团队使用的对照表
- **文档是最好的 Harness** — AGENTS.md/SOUL.md 的质量直接决定 Agent 行为质量
- **限制解空间反而提升效率** — 给 Agent 越清晰的约束，它越快收敛
- **Agent 看不到的信息等于不存在** — 一切重要知识必须存在于 workspace 内

## 十二、理解 Claude 的 Agentic 生态：把零散能力组织成可持续的工作流 ⭐⭐

**来源**：https://zhipu-ai.feishu.cn/wiki/OumbwSQgdiKnZjkrNSMcsQs7nue
**最后更新**：2026-03-31

### 核心要点

**五层组件模型**：

| 组件 | 定位 | 持久性 | 适用场景 |
|------|------|--------|---------|
| Prompts | 当下指令 | 即时 | 一次性需求、迭代优化、临时格式 |
| Skills | 可复用操作手册 | 跨会话 | 组织级工作流、领域型能力、个人偏好 |
| Projects | 长期工作空间 | 持续积累 | 持久上下文、工作空间组织、团队协作 |
| Subagents | 独立执行单元 | 任务级 | 任务专精、上下文管理、并行处理、工具隔离 |
| MCP | 连接层协议 | 基础设施 | 外部数据、业务系统、开发环境、自研系统 |

**Skills 渐进式加载**：
1. 元信息（~100 tokens）→ 判断"有没有用"
2. 完整说明（≤5k tokens）→ 了解"怎么做"
3. 脚本/资源文件 → 真正用到时才加载

**关键区分**：
- **Skills vs Subagents**：Skills 教"怎么做"（跨 agent），Subagents 解决"谁来做"（隔离执行）
- **Projects vs Skills**：Projects 提供上下文（"知道什么"），Skills 教导方法（"怎么做"）
- **Prompts vs Skills**：Prompts 即时不持久，Skills 跨会话可复用

### 详细笔记

**Projects 的核心特性**：
- 每个项目有独立聊天记录与知识库
- 200K context window
- 知识接近上限时自动启用 RAG（扩展至 10 倍）
- 适合为特定工作场景提供持续上下文

**Subagents 的使用场景**：
- 任务专精：代码审查、测试生成、安全审计
- 上下文管理：主对话保持主线，细碎工作下放
- 并行处理：多个 subagent 同时处理不同维度
- 工具隔离：限定为更安全的操作范围（如只读权限）

**如何选择组件**：
- 需要即时灵活 → Prompts
- 需要稳定重复 → Skills
- 需要持久上下文 → Projects
- 需要隔离执行 → Subagents
- 需要外部连接 → MCP

### 对 OpenClaw 的借鉴价值
- ⭐⭐
- 五层组件模型与 OpenClaw 的架构高度对应：Prompts↔对话、Skills↔Skills、Projects↔workspace、Subagents↔subagents、MCP↔plugins/MCP
- **渐进式加载**的理念可以优化 OpenClaw skill 的上下文管理
- **"Skills 跨 agent，Subagents 隔离执行"**的区分对 OpenClaw 的 skill/subagent 设计有直接指导意义
- Projects 的 RAG 模式（200K→10x）值得参考

**最后更新**：2026-03-31

### 核心要点

1. **三组件**：名称（清晰标识）+ 描述（触发场景）+ 指令（执行步骤）
2. **描述四要素**：能力 + 触发条件 + 上下文 + 边界
3. **低效 vs 高效描述对比**：
   - 低效："该 Skill 可处理 PDF 和文档相关事务"
   - 高效："全面的 PDF 处理工具，支持文本/表格提取、创建新 PDF、合并/拆分文档及表单填写。当 Claude Code 需要批量处理或分析 PDF 文档时触发。不支持简单查看或基础格式转换。"
4. **"菜单式"设计**：主文件=菜单，子文件=菜谱，按需加载避免臃肿
5. **测试三场景**：常规 → 边界（异常输入）→ 超出范围（不应触发）
6. **黄金法则**：做过 5 次 + 未来 10 次 → 值得做 Skill
7. **把"好"的标准写进指令**：结构标准、格式规范、验证规则、质量阈值

### 详细笔记

**指令推荐结构**：
1. 概述（简要介绍核心功能）
2. 前置条件（使用前需满足的条件）
3. 执行步骤（每步的输入输出）
4. 示例（具体使用场景）
5. 错误处理（常见错误及解决方法）
6. 局限说明（无法实现的功能）

**Skill 唤醒机制**：
- 不是关键词匹配，而是语义理解
- 需注意三个问题：
  - 拒绝模糊描述 → Claude 可能找不到对应的 Skill
  - 避免"一呼百应" → 多个 Skill 同时激活
  - 避免"叫天天不应" → 描述遗漏关键用例导致无法触发

**真实案例：Word 文档处理 Skill 的关键设计**：
- **强制性指令**：`MANDATORY - READ ENTIRE FILE`，防止 AI "偷懒"忽略细节
- **模块化引用**：主文件引用 docx-js.md 和 ooxml.md，完美实践"菜单式思路"
- **正反示例教学**：用 BAD 和 GOOD 代码对比，比抽象描述有效
- **错误预判**：批处理策略（3-10 个改动打包）、行号定位警告、代码风格指南

### 对 OpenClaw 的借鉴价值
- ⭐⭐
- **描述四要素**可以直接应用到 OpenClaw skill 的 description 字段优化
- **菜单式设计**原则应全面推广到 OpenClaw 的臃肿 skill 瘦身
- **测试三场景**可以作为 OpenClaw skill 测试的标准流程
- **把"好"的标准写进指令**的理念对 OpenClaw 的 coding-agent 等 skill 有直接参考价值
- **Skill-Creator 工具**的思路已在 OpenClaw 内置为 skill-creator 技能

## 十四、Superpowers + GLM：让 AI 像高级架构师一样写代码 ⭐

**来源**：https://zhipu-ai.feishu.cn/wiki/U2l7wd0Egi1hBnkUemQc8kign5b
**原始来源**：微信公众号（作者：冀博）
**最后更新**：2026-03-22

### 核心要点

**核心观点**：AI 编程下半场拼的是"控制力"，不是模型能力。

**Superpowers 是什么**：GitHub 95k+ Stars 的开源项目，给 AI 编程注入"工程灵魂"——剥夺 AI "直接写代码"的权力，强制三步流程：

1. **`/brainstorm`（头脑风暴）** — 拒绝盲目开工，先懂需求再动手。主动反问核心问题，倒逼需求明确
2. **`/write-plan`（制定计划）** — 输出 plan.md 施工图：Proposed Changes + Architecture Impact + Testing Strategy。没有确认不准动代码
3. **`/execute-plan`（TDD 执行）** — 强制 Red-Green-Refactor 循环：先写测试报错 → 写实现通过 → 优化重构。100% 测试覆盖

**Vibe Coding 的三大问题**：
- 需求没说清就开写，返工率拉满
- 改一个 Bug 带出三个新 Bug
- 一行测试都没有，上线即踩坑

**实测对比**：

| 维度 | Vibe Coding | Superpowers + GLM |
|------|------------|-------------------|
| 需求理解 | 猜着写，返工率高 | 强制对齐需求，零偏差 |
| 代码质量 | 随机波动 | 结构化清晰，符合工程规范 |

**GLM 配置**：
- Claude Code + GLM-4.7 / GLM-5
- Base URL: `https://open.bigmodel.cn/api/anthropic`
- 推荐使用 Coding Tool Helper（`npx @z_ai/coding-helper`）

### 详细笔记

**Superpowers 的价值不仅是工具**，更是工程方法论的落地：
- 将"架构师思维"强制注入 AI 编码流程
- TDD 循环确保代码可测试、可维护
- Plan 确认机制防止"乱改代码"

**与 Harness 工程的呼应**：
- Superpowers 的三步流程本质上是个人层面的 Harness
- 上下文工程（brainstorm）→ 架构约束（plan）→ 验证回路（TDD）

### 对 OpenClaw 的借鉴价值
- ⭐
- **三步流程**可以整合到 OpenClaw 的 coding-agent skill：brainstorm → plan → TDD execute
- **"控制力 > 模型能力"**的核心观点与 Harness 工程一致
- **TDD 循环**可以作为 OpenClaw 开发闭环的一部分
- GLM + Claude Code 的配置方式对 OpenClaw 用户有参考价值

### 关键洞察

1. **Harness 才是关键变量** — "相同模型，不同 Harness，结果天差地别"，OpenClaw 的提升空间在系统层而非模型层
2. **文档是最好的 Harness** — 所有重要知识必须存在于 workspace 内，Agent 看不到的信息等于不存在
3. **限制解空间反而提升效率** — 给 Agent 越清晰的约束，它越快收敛
4. **上下文工程是基础** — 静态上下文（文档/规则）+ 动态上下文（日志/CI），代码仓库是唯一事实来源
5. **Vibe Coding 的解药是控制力** — 不是更强的模型，而是更好的流程约束（brainstorm → plan → TDD）
6. **Agent 使用成熟度**：任务输入 → 规划 → 规则沉淀 → 环境配置 → 执行验证 → 工具接入 → 工作流自动化
