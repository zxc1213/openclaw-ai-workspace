## 一、认证授权服务（实战案例）⭐⭐⭐

**来源**：https://zhipu-ai.feishu.cn/wiki/YsQkwfmUAixTuskB1bUc3Xiun7f
**原始来源**：CSDN（作者：wanyj）
**GitHub**：https://github.com/userwanyong/auth-service
**最后更新**：2026-03-25

### 核心要点

1. **GLM + Claude Code 从零搭建认证授权微服务**：28 HTTP 接口 + 10 RPC 接口，一周工作量压缩到一天
2. **关键做法**：AI 先检索开源项目 → 制定方案 → 人工审核 → 编码 → 测试修复
3. **技术栈**：Java + MyBatis + JWT + Redis + Nacos + Dubbo + 原生 HTML/CSS/JS
4. **从单应用到多租户**：渐进式开发，每阶段人工审核后进入下一步

### 详细笔记

**开发流程**：

**阶段一：基础认证服务**
- 提示词："我要做一个有关登录、权限校验的微服务，请你检索市面上的相关服务"
- AI 通过网页搜索、开源项目分析，给出详细开发计划
- 遇到问题：默认使用 JPA 而非 MyBatis → 手动修正
- 测试结果：大部分功能成功，部分失败/漏写 → 将测试结果反馈 AI 修复

**阶段二：优化**
- JWT 无状态问题 → Redis 黑名单方案
- 过滤器校验失败无返回 → 增加具体错误信息

**阶段三：RPC 接口**
- 提示词："请你在保留 REST API 的同时，提供 RPC 接口并注册到 Nacos"
- AI 生成方案 → 审核 → 执行
- 遇到 javax vs jakarta 包版本冲突 → 通过 MCP（webSearchPrime + webReader）定位问题

**阶段四：多租户**
- 新开会话，独立实现
- 使用 3 个 MCP（webSearchPrime、webReader、开源仓库搜索）分析开源项目方案
- 审核 → 执行 → 测试 → 通过

**阶段五：前端管理界面**
- 原生 HTML/CSS/JS（轻量级，不引入框架）
- 与后端一起启动部署

**阶段六：部署与对接**
- Docker 镜像打包 → 阿里云镜像仓库
- 对接两个现有项目（修改 36 文件，新增 1 文件）

**提示词策略**：
- 初期：简单广泛，让 AI 自由发挥
- 后期：精准明确，"请你分析整个项目，确保充分理解"
- 关键：每次人工审核方案后再执行

### 对 OpenClaw 的借鉴价值
- ⭐⭐⭐
- **提示词策略**（初期宽松 → 后期精准）是 AI 编程的有效模式
- **多 MCP 协同**（搜索 + 读取 + 仓库分析）的实战效果验证了 MCP 的价值
- **渐进式开发**（基础 → 优化 → RPC → 多租户 → 前端 → 部署）每阶段独立会话，避免上下文过长
- **测试驱动的修复循环**（测试 → 反馈 → AI 修复 → 再测试）值得推广
- **一周压缩到一天**的效率提升非常震撼

## 二、Qclaw 自动化周报生成器 ⭐⭐

**来源**：https://zhipu-ai.feishu.cn/wiki/ZmxjwvkpriazVikrx51cPPeynCt
**原始来源**：CSDN（作者：摘星）
**最后更新**：2026-03-25

### 核心要点

1. **Qclaw + GLM 打造全自动周报系统**：从工作记录到周报自动生成
2. **Skill 驱动**：创建专门的周报生成 Skill，包含 SKILL.md + 模板文件
3. **定时 + 邮件分发**：支持 cron 定时任务和自动邮件发送
4. **多维度进阶**：支持多维度周报、数据可视化、多渠道分发、周报+OKR 联动

### 详细笔记

**Qclaw 配置 GLM**：
- Base URL: `https://open.bigmodel.cn/api/paas/v4`
- 模型：GLM-5 或 GLM-4.7

**Skill 目录结构**：
```
skills/
└── weekly-report/
    ├── SKILL.md          # 技能定义文件
    ├── work_log.txt      # 工作记录模板
    └── report_output/    # 报告输出目录
```

**SKILL.md 设计要点**：
- 明确技能定位：周报自动生成
- 输入：工作记录（文本格式）
- 输出：结构化周报（Markdown）
- 处理逻辑：自动分类、总结提炼、格式化输出

**使用方式**：
- 微信远程使用
- Qclaw 客户端直接使用
- 定时自动生成 + 邮件分发

**进阶玩法**：
1. 多维度周报（按项目/任务类型/时间维度）
2. 周报数据可视化（图表展示）
3. 多渠道智能分发（邮件 + 飞书 + 企业微信）
4. 周报 + OKR 联动（自动对齐目标进度）

### 对 OpenClaw 的借鉴价值
- ⭐⭐
- **Skill 驱动的自动化工作流**思路可直接应用到 OpenClaw
- **定时任务 + 邮件分发**的自动化模式可以参考
- **周报+OKR 联动**是值得 OpenClaw 探索的方向
- 整体思路可借鉴到 lark-workflow-standup-report skill

## 三、Claude Code 可交互学习网站 ⭐⭐

**来源**：https://zhipu-ai.feishu.cn/wiki/BcRgwVx7Ii7NT9kOzyfc5w3gn1f
**原始来源**：codefather.cn（作者：wanyj）
**在线体验**：https://cc.wanyj.cn
**GitHub**：https://github.com/userwanyong/claude-code-interactive-tutorial
**最后更新**：2026-03-24

### 核心要点

1. **解决 Claude Code 学习门槛**：命令行工具对新手不友好，用可交互网站降低门槛
2. **Trae Solo 双模式对比**：SOLO Builder（全自动）vs SOLO Coder（细粒度控制）
3. **完整产品闭环**：需求调研 → PRD/PAD → 前端实现 → 测试验收 → 迭代优化
4. **26 个标准命令**的渐进式引导学习路径

### 详细笔记

**环境准备**：
- Trae Solo + GLM-4.7
- Context7（最新文档神器）
- Firecrawl（网页抓取）
- ui-ux-pro-max（界面设计）
- edgeone-pages-mcp（部署）
- webSearchPrime + webReader + zread MCP

**SOLO Builder vs SOLO Coder 对比**：

| 维度 | SOLO Builder | SOLO Coder |
|------|-------------|------------|
| 控制粒度 | 粗（预制流程） | 细（每步可控）|
| MCP 支持 | ❌ | ✅ |
| Skills 支持 | ❌ | ✅ |
| 人工干预 | 少 | 多 |
| 适合阶段 | 快速原型 | 精细开发 |

**开发流程**：
1. 提示词 → 2. 确认需求 → 3. 生成 PRD/PAD → 4. 代码开发 → 5. 测试验收 → 6. 优化迭代

**SOLO Builder 的局限**：
- 不支持 MCP 和 Skills
- 无法做精细控制
- 初版质量有限（命令只支持 8 个）

**SOLO Coder 的优势**：
- 支持 Context7 获取最新文档
- 支持自定义 MCP 工具
- 控制力度更细
- 最终版本实现了手动命令输入 + 26 个标准命令 + 打字机式引导 + 进度追踪

### 对 OpenClaw 的借鉴价值
- ⭐⭐
- **SOLO Coder > SOLO Builder**的结论验证了 OpenClaw 的设计方向（细粒度控制 > 全自动黑箱）
- **Context7 + MCP** 的组合是获取最新文档的标准方案
- **渐进式学习路径**的设计思路可用于 OpenClaw 的用户引导
- **多工具协同**（Context7 + Firecrawl + UI 设计 MCP + 部署 MCP）的实战模式值得参考
