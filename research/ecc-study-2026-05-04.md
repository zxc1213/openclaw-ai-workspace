# ECC (Everything Claude Code) 深度研究笔记

**日期**: 2026-05-04
**研究对象**: https://github.com/affaan-m/everything-claude-code
**研究目的**: 学习核心 skill 实现模式，评估可移植性，指导 OpenClaw skill 体系升级

## 一、研究的 Skills 概览

### 1. strategic-compact — 战略性上下文压缩

**核心机制**: 在逻辑断点（非任意自动触发）建议 `/compact`，避免 mid-task 丢失关键上下文。

**关键设计**:
- 使用 PreToolUse hook（Edit/Write）追踪工具调用次数
- 默认阈值 50 次工具调用后首次提醒，之后每 25 次提醒
- 提供**阶段转换决策表**：Research→Planning（Yes）、Planning→Implementation（Yes）、Mid-implementation（No）等
- 明确**压缩后存活 vs 丢失**的内容清单（CLAUDE.md 存活、文件内容丢失）

**与我们的对比**:
- OpenClaw 有自动 compaction（`/compact`），但没有 strategic-compact 的逻辑断点意识
- 我们的 AGENTS.md 有简单的上下文管理策略，但缺乏具体的阶段转换指导
- **可移植价值**: 高 — 阶段转换决策表和"写后压缩"原则直接适用

**关键洞察**:
> "The investigation itself creates context that changes the output." — 这不是只读的，而是通过 hook 强制执行调查行为

### 2. continuous-learning (v1 + v2) — 持续学习

**核心机制**: 
- v1: Stop hook 在会话结束时提取模式，保存到 `~/.claude/skills/learned/`
- v2 (Instinct-based): PreToolUse/PostToolUse hook 捕获观察 → 后台 agent 分析 → 创建原子 "instinct"（带置信度 0.3-0.9）

**v2 的关键创新**:
- **Instinct 模型**: 原子行为单位，每个有 trigger/action/confidence/domain/evidence
- **项目隔离**: git remote URL 哈希作为项目 ID，防止跨项目污染
- **进化路径**: Instincts → cluster → skills/commands/agents
- **置信度衰减**: 被反驳的 instinct 降低置信度
- **Promotion**: 在 2+ 项目中出现的 instinct 提升为全局

**与我们的对比**:
- 我们有 `self-improvement` skill 和 daily notes 的 `[self-reflection]` 标记
- 我们的 SOUL.md 有"主动记忆"机制，但缺乏系统化的自动提取
- **可移植价值**: 中高 — instinct 模型和置信度评分概念值得借鉴，但 v2 依赖 Claude Code 的 hook 系统，OpenClaw 需要适配

### 3. gateguard — 事实强制门控

**核心机制**: PreToolUse hook 阻断 Edit/Write/Bash，强制 Claude 在操作前收集具体事实。

**三阶段门控**:
1. **DENY** — 阻断首次操作尝试
2. **FORCE** — 精确指定需要收集哪些事实
3. **ALLOW** — 事实呈现后允许重试

**效果**: A/B 测试显示 +2.25 分（9.0 vs 6.75）的质量提升

**关键洞察**:
> "LLM self-evaluation doesn't work. Ask 'did you violate any policies?' and the answer is always 'no.'"

**与我们的对比**:
- 我们的 AGENTS.md 有"高危操作前自检"，但不够系统化
- **可移植价值**: 高 — "强制调查而非自我评估"的理念可以融入 skill-vetter 和 subagent 模板

### 4. context-budget — 上下文预算审计

**核心机制**: 扫描所有组件（agents/skills/rules/MCP servers/CLAUDE.md），估算 token 消耗，产出优化建议。

**关键设计**:
- 分类: Always needed / Sometimes needed / Rarely needed
- 检测: 膨胀描述（>30 words）、重型 agent（>200 lines）、冗余组件、MCP 过度订阅
- MCP 是最大的开销杠杆（每个 tool schema ~500 tokens）

**与我们的对比**:
- 我们有 `zai-token-stats` skill，但主要关注用量统计而非预算审计
- **可移植价值**: 中 — 87+ skills 的审计对我们很有意义，可以增强 zai-token-stats 或创建独立 skill

### 5. skill-stocktake — 技能盘点

**核心机制**: 扫描所有 skills，用 AI 判断每个 skill 的处置（Keep/Improve/Update/Retire/Merge）。

**关键设计**:
- Quick Scan（仅检查变更的 skills）vs Full Stocktake（全部审查）
- 使用 subagent 批量评估，每批 ~20 个
- 结果缓存到 `results.json`，支持断点续评
- 评估维度: Actionability / Scope fit / Uniqueness / Currency
- 原因质量要求很高（不能用"unchanged"等模糊描述）

**与我们的对比**:
- 我们的 `skill-vetter` 侧重安全审计，不评估质量/有效性
- **可移植价值**: 高 — 可以与 skill-vetter 互补，增加质量评估维度

### 6. configure-ecc — ECC 安装配置器

**核心机制**: 交互式安装向导，引导用户选择 skill 类别并安装。

**关键设计**:
- 分层安装: User-level (~/.claude/) vs Project-level (.claude/)
- 7 个可选类别，45+ skills
- 每个类别有详细描述表格

**可移植价值**: 低 — 我们已经有完善的 skill 管理机制

## 二、ECC Skill Frontmatter 规范分析

ECC 使用的 frontmatter 格式（从多个 SKILL.md 提取）：

```yaml
---
name: skill-name
description: 一句话描述
origin: ECC | community
version: x.y.z  # 部分有
---
```

**观察**:
- `name` 和 `description` 是标配
- `origin` 标注来源（ECC/community）
- `version` 仅部分 skill 有（如 continuous-learning-v2 有 version: 2.1.0）
- 没有 `tags` 字段
- 没有 `trigger` 或 `when-to-use` 的结构化字段（这些放在正文 `## When to Activate` 中）

## 五、AGENTS.md Subagent 模板增强建议

在现有 Constraints 中增加 verification loop：

```markdown
5. **自检清单**：任务完成后必须验证
   - [ ] 产出文件存在且格式正确
   - [ ] 代码可运行/可编译（如适用）
   - [ ] 未引入新的 lint 错误
   - [ ] 未遗漏 Done 标准中的任何条目
6. **修正循环**：自检发现问题 → 修正 → 再验证（最多 2 轮）
7. **异常检测**：
   - 输出截断（看到 [... N more characters truncated]）→ 用 offset/limit 重读
   - 工具调用失败超 2 次 → 记录并报告，不无限重试
   - 上下文接近上限（>60%）→ 考虑分批处理
```

## 六、Skill-Vetter 扩展建议

参考 gateguard 的安全理念，建议增加：

### 6.1 Frontmatter 完整性检查（Low, -3）
```yaml
check: frontmatter-completeness
rules:
  - 缺少 name → -3
  - 缺少 description → -3
  - description > 100 字符 → 标记为 verbose
```

### 6.2 敏感信息检测增强
现有 `plaintext-secrets` 已覆盖主要 API key 格式。建议增加：
- 硬编码私有路径（如 `/home/rays/`、`/Users/username/`）→ Medium
- 硬编码 IP 地址（非 127.0.0.1/localhost）→ Medium

### 6.3 工具权限越界检测
- Skill 指示 agent 执行超出正常 skill 范围的操作（如删除系统文件）→ High

## 七、研究总结

ECC 的核心价值在于：
1. **Hook 驱动**的行为增强（而非纯 prompt 指令）
2. **量化证据**（A/B 测试、token 估算）支撑设计决策
3. **分层抽象**（instinct → skill → agent → command）的学习进化路径
4. **系统性审计**（skill-stocktake、context-budget）而非一次性检查

我们的优势：
1. OpenClaw 的 skill 系统更成熟（87+ skills，完善的目录规范）
2. skill-vetter 的安全审计比 ECC 更系统化
3. AGENTS.md 的 subagent 派发规范更完善

改进方向：
1. 增加"逻辑断点感知"的上下文管理
2. 强化"强制调查"而非"自我评估"的安全理念
3. 建立 skill 质量评估机制（不只是安全）
4. 考虑引入 instinct-based 学习模型（长期）
