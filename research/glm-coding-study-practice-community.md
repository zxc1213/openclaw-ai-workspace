## 板块一：官方最佳实践

### 1.1 企业 AI Native 产研转型范式（1.0）

**来源**：https://zhipu-ai.feishu.cn/wiki/VD9PwZHDmiKi22krJYvc444snjc

**核心要点**：
1. 工程师写代码时间仅占20%，单纯提升编码效率只是单点优化，AI Native 的最大价值在于理解用户意图、将需求转化为完整产品方案
2. 传统三大痛点：效率瓶颈（复杂逻辑依赖资深工程师）、工具碎片（各环节自动化难贯通）、协作割裂（信息孤岛+上下文丢失）
3. AI Native 转型价值：研发周期从天级降至小时级、产研一体化协作、创新节奏从季度级升至月度级
4. 智谱三大核心能力：卓越代码生成（GLM-4.6, 355B参数, 200K上下文）、强大Agent协同（多Agent编排）、企业级定制（私有化部署）

**详细笔记**：

文章强调了一个关键认知转变——AI Native 不只是「用AI写代码更快」，而是从根本上重构产研团队的工作方式：

- **超越编码**：工程师日常只有约20%时间在写代码，还有80%在沟通、理解需求、查文档、调bug等。只优化编码效率只抓住了冰山一角
- **从"代码驱动"到"需求驱动"**：AI理解用户意图，将需求转化为完整产品方案，代码回归为实现手段
- **产研一体化**：通过统一的AI工具贯穿需求→设计→编码→测试→部署全链路，上下文不再在环节间丢失

方案架构分三层：
- 底层（模型基座）：大模型 + 私有知识库，深度理解企业代码库和业务逻辑
- 中层（AI产研基础设施）：统一工具平台，上下文记忆与协同机制
- 上层（研发工具适配）：嵌入IDE、Git、Jira等日常工作流

**对 OpenClaw 的借鉴价值**：
- OpenClaw 的定位与"AI Native 基础设施"高度吻合——我们就是那个「中层」
- 强调「上下文贯穿全链路」验证了我们 Skills + MCP + Subagents 体系的设计方向
- 企业级私部署需求是我们可以拓展的方向

### 1.3 需求阶段：智能需求生成与分析（2.1）⭐⭐⭐

**来源**：https://zhipu-ai.feishu.cn/wiki/QfymwjVvgiBgngki5h0cGJM1nBb

**核心要点**：
1. AI Native 需求阶段 = 问题对齐→信息收集→机会空间→方案共创→最小验证→PRD生成→设计稿联动，7步闭环流水线
2. PM角色转变：从「写PRD的人」变成「问题定义者&审核者」，AI变成「调研与推理引擎」
3. 4类AI验证方式：模拟用户访谈、生成交互草图+原型、日志回放假设检验、Prompt级MVP
4. PRD结构包含AI参与点（模型、Agent、工具调用、数据依赖），这是与传统PRD的关键差异

**详细笔记**：

这是全系列最有实操价值的一篇。智谱以自身10万用户产品为背景，给出了完整的PM+AI协作流程：

**Step 1 问题对齐**：输入业务目标→AI拆解为可度量指标→识别信息缺口→输出《问题定义One-pager》

**Step 2 信息收集**：AI做「Research Ops」——汇总埋点数据、工单、反馈社区、竞品文档等多源信息，聚类用户痛点、识别高频场景、提炼「影响指标」的关键路径

**Step 3 机会空间**：AI基于痛点和约束自动生成「机会地图」，每个机会包含目标用户、场景、痛点、潜在方案、预期指标提升

**Step 4 方案共创**：为每个机会生成3类方案（极简MVP版、体验优先版、AI增强版），包含核心价值叙述、用户旅程、功能点拆分

**Step 5 最小验证**：4种验证方式可组合使用
- AI模拟用户访谈：AI扮演不同用户画像对方案质疑
- 生成交互草图+可点击原型
- AI驱动日志回放：喂匿名化行为日志让AI分析需求迹象
- Prompt级MVP：不改代码，通过指令模拟方案让种子用户试用

**Step 6 PRD自动生成**：AI从前面所有产出物自动拼装PRD初稿，PM做最终审核。关键新增：**PRD结构中必须包含"AI参与点"——说明哪些环节用了模型、Agent、工具调用、数据依赖**

**Step 7 设计稿联动**：需求结束时同时产出PRD + Wireframe/设计Brief，两者源自同一AI协作链路

**角色分工设计**：
- Research Agent：搜集、归纳信息
- Product Writer Agent：生成PRD
- UX Agent：生成页面结构与流程
- Analyst Agent：设计和解读验证

**对 OpenClaw 的借鉴价值**：
- 这套7步流程可以直接转化为OpenClaw的Workflow Skill
- 「AI参与点」在PRD中的标注方式，启示我们可以为SOUL.md/CLAUDE.md增加类似的「能力声明」结构
- 多Agent分工模式（Research/Writer/UX/Analyst）与我们的Subagent架构完美对应
- 最小验证的4种方式都可以用MCP+Skill实现

### 1.4 技术方案：AI辅助系统方案与架构设计（2.2）⭐⭐

**来源**：https://zhipu-ai.feishu.cn/wiki/QGLvwViGWi4H9Qk0ukNcuBzNnkh

**核心要点**：
1. AI辅助技术方案设计三步走：需求洞察与关键决策→技术选型→系统设计（DDD微服务拆分）
2. AI能识别需求中的「模糊点」并列出需要架构师确认的关键决策点（如租户隔离级别、数据时效性、性能目标等）
3. 技术选型时AI会优先考虑利用企业现有技术栈，只在必要时引入新中间件并附必要性分析
4. 系统设计输出包括：微服务拆分（领域驱动）、关键交互流程、OpenAPI规范的接口定义

**详细笔记**：

文章以「多租户SaaS知识库平台」为例，完整演示了AI辅助架构设计流程：

**AI的关键能力**：
- 自动将需求分解为功能性/非功能性要求
- 识别设计模糊点并提出决策选项
- 基于企业现有技术栈做推荐（Java 21/Spring/Flink/Kafka/MySQL/PG+pgvector/ES）
- 领域驱动设计的微服务拆分（6个服务+API网关）
- 生成完整的交互流程（文档处理流水线、智能问答流程）
- 输出OpenAPI 3.0规范的接口定义

**关键决策识别示例**：
- 租户隔离：共享DB+tenant_id vs 独立DB
- 数据时效：实时 vs 异步
- LLM集成：绑定单一模型 vs 可插拔架构
- 检索策略：纯向量 vs 混合检索

**对 OpenClaw 的借鉴价值**：
- AI识别「模糊点」的能力可以集成到我们的架构咨询Workflow中
- 「优先利用现有技术栈」的原则应该在Skill中固化
- 微服务拆分+交互流程+API定义的完整输出格式可作为模板

### 1.6 测试阶段：端到端的全流程自动化测试（2.4）⭐

**来源**：https://zhipu-ai.feishu.cn/wiki/WDL5wqz9xi067bkH0Etcb1LUnEd

**核心要点**：
1. AI驱动测试的两个核心场景：接口压力测试全流程自动化、端到端自动化测试
2. 接口压测流程：prompt描述→AI生成脚本→自动调试→解决异常→再验证→生成报告，全程无需人工
3. 端到端测试：需求分析→生成测试用例→生成自动化代码→执行并生成报告，可节省50-70%人力
4. 关键特点：AI生成的测试代码具备主动处理异常的能力，提高了脚本健壮性

**详细笔记**：

**接口压测案例**：
- 用自然语言描述压测需求（含请求示例和结果样例）
- AI自动生成调试脚本和readme
- AI自行调试脚本并解决错误
- 自动生成压测报告
- 时间成本：天→分钟，节约90%

**端到端测试案例**（以智谱官网为例）：
- 通过prompt描述任务+图片分析MCP工具→生成测试用例→结合selenium框架生成代码
- AI自动拆分任务、按步骤执行、自动处理异常

**对 OpenClaw 的借鉴价值**：
- AI驱动的压测流程可以作为自动化测试Skill的设计参考
- 「主动调试+异常处理」能力是测试Agent的关键差异化能力
- 可以考虑为OpenClaw增加自动化测试相关的Subagent

### 1.7 对话式运维升级：以大模型打造自助排障能力（2.5）⭐⭐

**来源**：https://zhipu-ai.feishu.cn/wiki/E6OrwYZari6AbykQHbscGJfcnsb

**核心要点**：
1. 传统运维三大痛点：工具割裂（上下文切换频繁）、知识分散（隐性经验依赖个人）、排查路径不标准（难复制传承）
2. 大模型运维三大新能力：语义检索（自然语言跨系统查询）、上下文整合（自动汇聚告警/指标/日志成事故故事线）、自助排查服务化（slash命令一键排障）
3. 实战案例：Claude Code + GLM-4.6 + Grafana MCP + 自定义support MCP，实现一条命令完成用户欠费问题全链路排查
4. 排查流程可封装为可复用的「slash命令+智能代理+工作流」，通过marketplace安装

**详细笔记**：

**语义检索示例**：
用户说「获取本服务在过去5分钟内接口QPS超过100的数据」，AI自动：
1. 找到Prometheus数据源
2. 构建正确的PromQL查询
3. 执行查询并格式化结果
4. 输出按QPS排序的接口统计

**自助排障实战**：
一个完整的客户问题排查案例——用户购买coding套餐但出现1113欠费错误。Claude Code + GLM-4.6 + 自定义MCP的完整排障流程：
1. 通过手机号查询用户信息（support MCP）
2. 查询1113错误日志记录
3. 查询用户购买的套餐信息（MySQL MCP）
4. 查询产品表结构（自动适应表结构差异）
5. 多次修正SQL查询（AI自动处理错误）
6. 给出最终结论：用户购买的是普通tokens包，不是coding套餐

整个流程中AI展现了：
- 自动参数类型修正（integer→string）
- SQL错误自修复（Unknown column→先查schema再修正）
- 跨多表关联分析
- 最终生成结构化排查报告

**对 OpenClaw 的借鉴价值**：
- **高价值**：对话式运维是我们的核心应用场景之一
- support MCP（查用户、查日志、查MySQL schema、执行SQL查询）可以直接参考来实现我们自己的运维MCP
- 「slash命令+marketplace安装」模式与OpenClaw的Skill体系完全一致
- `/logerror + 日志ID` 这种标准化排障入口非常值得借鉴
- 排查流程模板（事故故事线构建）可以作为运维Skill的参考

### 2.1 一套教程让你不再畏惧使用AI Coding ⭐⭐

**来源**：https://zhipu-ai.feishu.cn/wiki/Bowiwy3ggiEb3RkvfG1c5FOxnEc
**作者**：冰冻大西瓜
**原文**：https://linux.do/t/topic/1114315

**核心要点**：
1. 选择Claude Code的核心原因：支持Sub-agents子代理，其他工具（codex/cursor/zed）暂无此功能
2. GLM-4.6等国内模型已兼容CC，解决了国内用户使用CC的支付和封号问题
3. 全局提示词应该精简（4要素：语言偏好、操作系统/shell环境、主动使用MCP、主动使用子代理），避免冗长矛盾
4. MCP推荐Exa MCP（互联网访问能力，比context7 MCP更省token）
5. 实战案例：VitePress博客 + Rough Notation动画库集成

**详细笔记**：

**工具选型逻辑**：
- Claude Code > Cursor/Codex/Zed，因为Sub-agents功能独一无二
- GLM-4.6 > Sonnet 4.5（性价比），国内可用且便宜
- 禁用CC自动升级（太频繁），改用`pnpm update -g`手动升级

**环境配置技巧**：
- 用PowerShell Profile设置环境变量（`$env:ANTHROPIC_AUTH_TOKEN`），不用系统环境变量（修改繁琐）
- 全局MCP配置在 `.claude.json`，全局提示词在 `.claude/CLAUDE.md`
- 状态栏插件 `@cometix/ccline` 可监控token消耗

**全局提示词精简原则**（值得注意）：
- 不要写500字的通篇大论提示词，token是有限的
- 避免矛盾指令（「既要灵活又要准确」让AI无所适从）
- 4个核心要素就够了

**对 OpenClaw 的借鉴价值**：
- Sub-agents 是 CC 的杀手锏，验证了我们的 Subagent 架构方向正确
- Exa MCP 的推荐值得了解——如果需要给OpenClaw增加互联网搜索能力
- 全局提示词精简原则直接适用于我们的 SOUL.md / AGENTS.md 设计

## 板块总结：核心知识脉络

### 官方最佳实践 — 三条主线

**主线一：企业转型认知**
- AI Native ≠ AI辅助编码，而是全流程重构（从需求到运维）
- 工程师写代码只占20%时间，全链路提效才是目标
- 组织竞争力从季度级创新升维到月度级

**主线二：研发流程AI化（7阶段）**
1. 需求阶段（最有价值）：PM+AI的7步闭环——问题对齐→信息收集→机会空间→方案共创→最小验证→PRD生成→设计稿联动
2. 技术方案：AI辅助架构设计——识别模糊点→技术选型→DDD微服务拆分→API定义
3. 开发阶段：前端代码智能化 + 场景化实践（待深入）
4. 测试阶段：AI生成压测脚本→自动调试→自动报告，节约90%时间
5. 运维阶段：语义检索+上下文整合+自助排查服务化（slash命令一键排障）

**主线三：能力架构设计**
- 模型层：GLM-4.6（355B参数、200K上下文）
- 工具层：多Agent编排、MCP工具链、Skills沉淀
- 组织层：PM变为问题定义者，AI变为调研推理引擎
